# Input Sufficiency And Challenge Readiness

## Behavior Change Thesis
When loaded for a candidate bundle that feels too thin, this file makes the model route to framing or targeted research instead of inventing challenge questions from missing decisions.

## When To Load
Load this when candidate synthesis might be too thin to challenge, evidence is missing, or the pass is drifting into blank-page design.

## Decision Rubric
- Challenge only after the bundle names a problem frame, candidate decisions, constraints, assumptions or open questions, and at least one seam whose answer could change planning.
- Return to framing when the problem, success criterion, or user-visible outcome is absent.
- Return to synthesis or research when the candidate path is a list of options rather than a chosen direction to pressure-test.
- Reopen one specialist lane when a single missing fact blocks challenge, such as mixed-version client behavior, tenant boundary evidence, or cache invalidation precedent.
- Do not ask "do we have enough detail?" Name the exact missing decision or evidence and the smallest route that would unblock challenge.

## Imitate
- "Which candidate decision would change if the retry assumption is false, and what evidence currently supports it?"
  - Copy the way the question ties sufficiency to a decision fork, not generic completeness.
- "Is the missing constraint product policy, repository fact, or specialist evidence?"
  - Copy the routing split: different missing facts need different next actions.
- "If no candidate decision is present, should this return to synthesis instead of challenge?"
  - Copy the refusal to turn challenge into primary design.

## Reject
- "What are all the open questions?"
  - Fails because it invites a broad intake list instead of a planning-risk challenge.
- "Can we write a better spec?"
  - Fails because this skill does not author the spec or decide the design.
- "Ask the user whether the repository supports mixed-version clients."
  - Fails when the fact should be researched from repo evidence or deployment docs instead of pushed to the user.

## Agent Traps
- Confusing this synthesis checkpoint with the later `spec-clarification-challenge` approval gate.
- Treating missing links as automatic failure even when the challenged assumption is low-impact and reversible.
- Filling missing architecture or product choices with plausible defaults instead of routing them.
- Producing a full discovery questionnaire when the orchestrator only needs to know whether challenge can proceed.
