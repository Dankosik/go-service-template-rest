# Engineering standard: колонночные и аналитические базы в Go‑микросервисах

## Область применения

Колонночные/аналитические СУБД (OLAP) оправданы, когда основной профиль нагрузки — чтение и агрегации по большим объёмам данных (сканирование, фильтрация, `GROUP BY`, time‑series/лог‑аналитика), где выигрывают колонночное хранение и компрессия. В фундаментальных работах по колонночным СУБД отмечается, что колонночное представление делает данные более «сжимаемыми» и позволяет существенно ускорять аналитические запросы за счёт эффективного чтения только нужных столбцов и работы с компрессированными данными. citeturn0search19turn0search31

OLTP и OLAP принципиально различаются по «единице доступа» и требованиям к консистентности. Для OLTP типичны точечные операции по строкам, частые `UPDATE/DELETE`, строгая транзакционность и немедленная согласованность. Для OLAP типичны «широкие» чтения и агрегации, а обновления часто реализуются через пере‑загрузку/пере‑индексацию или «имитацию апдейта» через добавление новой версии события и последующие фоновые процессы. Это явно проявляется в документации и поведении популярных аналитических систем: например, в ClickHouse подчёркивается ориентация на append‑only и eventual consistency, а не на немедленную сильную консистентность в стиле OLTP. citeturn4view0turn9search2turn14search1

image_group{"layout":"carousel","aspect_ratio":"16:9","query":["row store vs column store illustration","columnar storage compression illustration","OLTP vs OLAP diagram row vs column"]}

Подход «аналитическое хранилище рядом с микросервисом» применяйте, когда сервису нужно: (а) отдавать аналитические витрины/отчёты/дашборды, (б) обеспечивать продуктовую аналитку (фаннелы, cohort, ретеншен), (в) обслуживать observability‑кейсы (логи/метрики/трейсы), (г) запускать тяжёлые расчёты (batch/near‑real‑time) с предагрегациями и длительным хранением. Для ClickHouse это типовые use cases (real‑time analytics, logs/events/traces), а для Druid — сценарии time‑based аналитики с сегментацией и ingestion‑пайплайнами. citeturn4view1turn10search8turn2search3

Когда этот подход не применять (или применять только за пределами «горячего» OLTP‑контурa микросервиса):

- Если микросервису нужна транзакционная модель (строгие инварианты, частые обновления/удаления, «прочитал‑проверил‑записал» в рамках одной транзакции). У ClickHouse «мутации» реализуются через `ALTER TABLE` и по смыслу являются дорогими операциями для аналитических нагрузок; ClickHouse отдельно рекомендует избегать частых/масштабных мутаций и использовать альтернативные движки/модели данных. citeturn9search2turn14search1  
- Если вы пытаетесь использовать аналитическое хранилище как «основную» БД для высокочастотных row‑level мутаций: в BigQuery прямо отмечаются rate limits для DML и необходимость backoff при большом числе jobs на одну таблицу; в материалах BigQuery также противопоставляют мутации в warehouse‑подходе и ограничения по частоте мутаций относительно OLTP. citeturn9search14turn9search16  
- Если продуктовые требования подразумевают «жёсткую» корректность на уровне отдельных событий «прямо сейчас», а выбранный движок даёт корректность с задержкой из‑за фоновых процессов. Например, `ReplacingMergeTree` в ClickHouse удаляет дубликаты только во время merge, а merge происходит асинхронно и не гарантирует отсутствие дублей в любой момент. citeturn1search1turn14search5

Нормативная позиция для template microservice: аналитическое хранилище по умолчанию не должно быть частью критического request path. Стандартный «boring» паттерн — OLTP/командный контур (транзакции) отдельно, аналитический контур — асинхронно через ingestion (batch/stream), с явной моделью идемпотентности и дедупликации. Эта рекомендация следует из архитектурных свойств систем: ClickHouse оптимизирует массовые insert’ы и допускает eventual consistency; Druid хранит данные в неизменяемых сегментах и управляет заменой данных на уровне сегментов; BigQuery тарифицирует обработанные байты и ориентирован на аналитические jobs. citeturn4view0turn10search8turn0search5

## Рекомендуемые дефолты для greenfield template

Ниже — «боевой минимум» для Go‑микросервиса, который должен работать с ClickHouse/BigQuery/Druid‑подобными аналитическими системами без догадок со стороны LLM.

### Базовый каркас data‑heavy интеграции

Рекомендуемый дефолтный контур в template repo:

1) **Immutable event model как исходник истины для аналитики.** События append‑only, с явными полями `event_time` (время события) и `ingest_time` (время приёма), а также `event_id`/`dedup_key` для идемпотентности. Такой подход напрямую согласуется с тем, как аналитические движки «эмулируют апдейты»: ClickHouse предлагает имитацию update через вставку нескольких версий и фоновые процессы (например `ReplacingMergeTree`), а Druid управляет заменами на уровне сегментов/интервалов. citeturn1search5turn10search8turn10search3

2) **Отдельный ingestion worker (или отдельный режим запуска сервиса).** Не смешивать HTTP‑API и тяжёлую загрузку в одном request path. Для ClickHouse прямо рекомендуется батчить insert’ы (порядка тысяч и выше) и при необходимости использовать async inserts; Druid строит ingestion вокруг задач/спеков и сегментов; BigQuery для стриминга предлагает Storage Write API и различает at‑least‑once vs exactly‑once семантики в зависимости от типа stream. citeturn4view0turn2search9turn2search1turn2search4

3) **Явные SLA на запросы и таймауты.** В Go это реализуется через `context.Context`; официальная документация Go подчёркивает использование контекста для отмены операций и досрочного завершения работы. citeturn13search2turn13search6

4) **Безопасность запросов: параметризация и запрет динамической сборки SQL из пользовательского ввода.** OWASP рекомендует prepared statements/parameterized queries как базовую защиту от SQL injection. citeturn7search3turn7search5

### Дефолты для ClickHouse

Документация entity["company","ClickHouse, Inc.","database company"] задаёт важные свойства ClickHouse как OLAP‑системы: оптимизация под высокую скорость вставки (миллионы строк/сек) достигается компромиссами по немедленной консистентности; ClickHouse оптимизирован под append‑only и даёт eventual consistency. citeturn4view0

Нормативные дефолты схемы и ingestion:

- **ENGINE:** начинать с MergeTree‑семейства (в проде часто `ReplicatedMergeTree`/шардирование — вне этого подраздела, но template должен оставаться совместимым). MergeTree хранит данные частями (parts), каждая часть сортируется по primary key; далее части мерджатся в фоне, и части разных partition не мерджатся между собой. citeturn12view1turn11view0  
- **PARTITION BY:** использовать как инструмент управления жизненным циклом, а не «первую кнопку ускорения запросов». В ClickHouse подчёркивается, что partitioning — прежде всего data management техника; она может улучшить или ухудшить запросы в зависимости от паттерна доступа, и требует осознанного выбора. citeturn11view0  
- **TTL/Retention:** если используется TTL для удаления старых данных, ClickHouse рекомендует выравнивать partition по тому же полю времени (день/месяц) — так ClickHouse может «ронять» целые партиции значительно эффективнее, чем удалять строки. citeturn0search4turn11view0  
- **ORDER BY / primary key:** ClickHouse хранит строки на диске упорядоченными по primary key; sparse primary index оперирует «гранулами» и позволяет пропускать данные, но primary key не обязан быть уникальным, и может быть много строк с одинаковым значением ключа. citeturn12view1turn18search14turn18search3  
- **Вставки:** по умолчанию батчить. ClickHouse рекомендует вставлять крупными батчами (минимум ~1000 строк, «идеально» 10k–100k) и поддерживать **консистентные батчи** для идемпотентных ретраев: если повторить insert с теми же данными в том же порядке, ClickHouse может автоматически дедуплицировать повтор. citeturn4view0turn2search17  
- **Идемпотентность:** при ретраях использовать «консистентные батчи» и/или явный `insert_deduplication_token`, если нужно управлять семантикой дедупликации вставок. citeturn4view0turn3search1turn3search13  
- **Апдейты/коррекции:** избегать частых мутаций (`ALTER … UPDATE/DELETE`) и особенно «принудительных» merge‑операций в ежедневном контуре. ClickHouse формулирует правило «avoid frequent or large-scale mutations» и рекомендует альтернативные движки (например `ReplacingMergeTree`, `CollapsingMergeTree`) для коррекции данных. citeturn14search1turn1search5turn1search1  
- **OPTIMIZE FINAL:** запрещать как «операцию по умолчанию» — ClickHouse объясняет, что `OPTIMIZE … FINAL` CPU/I/O‑дорогая операция (декомпрессия → merge → компрессия → запись) и её не следует использовать как рутинный механизм. citeturn14search0turn14search9  
- **Pre-aggregation:** использовать materialized views, чтобы переносить вычисления на insert time. ClickHouse описывает инкрементальные materialized views: новая вставка в исходную таблицу автоматически применяет запрос MV и пишет результат в целевую таблицу; далее частичные результаты мерджатся. citeturn1search0turn1search4

Go‑интеграция (дефолты template):

- Использовать официальные Go‑клиенты: ClickHouse отмечает два «официальных» клиента, где `clickhouse-go` поддерживает и `database/sql`, и native API, а `ch-go` — низкоуровневый native‑клиент. citeturn2search2  
- Для высокочастотных insert’ов иметь отдельный слой batching с контролем размера батча/времени флаша согласно рекомендациям ClickHouse по batching/async inserts. citeturn4view0turn16view0  
- Контекст/таймауты обязательны (см. правила LLM ниже): отмена запроса через context — базовый механизм в Go. citeturn13search2turn13search6

### Дефолты для BigQuery

BigQuery (в составе платформы entity["company","Google Cloud","cloud platform"]) — fully‑managed аналитический warehouse с моделью стоимости «за обработанные данные». В pricing‑документации подчёркивается: запрос тарифицируется по объёму обработанных данных в выбранных столбцах даже при наличии `LIMIT`, а partitioning/clustering — практики, уменьшающие объём сканирования и стоимость. citeturn0search5

Нормативные дефолты таблиц/запросов:

- **PARTITION BY:** использовать partitioned tables и закладывать partition pruning как обязательный способ контроля стоимости/латентности; BigQuery описывает pruning: при qualifying filter по partition column BigQuery сканирует только подходящие партиции. citeturn0search1turn13search4  
- **require_partition_filter = TRUE:** включать по умолчанию в greenfield, чтобы защититься от случайных full scan; BigQuery прямо говорит, что обязательный partition filter снижает стоимость и улучшает производительность, и приводит пример `require_partition_filter = TRUE`. citeturn6view0turn13search4  
- **Правильные фильтры для pruning:** BigQuery даёт конкретные best practices — использовать **константные** фильтры и изолировать partition column; динамические выражения и фильтры, зависящие от нескольких полей, могут привести к сканированию всех партиций. citeturn13search4  
- **CLUSTER BY:** использовать clustered tables для block pruning: BigQuery сортирует данные по clustering columns и разбивает на блоки — это позволяет пропускать блоки и уменьшать обработку. citeturn1search2turn15view0  
- **Правильные фильтры для clustering:** BigQuery предписывает фильтровать clustering columns в порядке сортировки; также прямо запрещает «оборачивать» clustered columns в функции или сравнивать clustered column с другой колонкой, потому что block pruning тогда не применяется. citeturn15view0  
- **Не шардировать «таблицами‑суффиксами» по датам:** в performance best practices BigQuery отдельно рекомендует избегать «слишком большого числа shards» и вместо date‑sharding использовать time‑partitioned tables. citeturn13search8

Ingestion и семантика доставки:

- **Streaming:** для новых проектов BigQuery рекомендует Storage Write API вместо legacy streaming API; Storage Write API описывает default stream (at‑least‑once) и application‑created streams для exactly‑once семантики через offsets. citeturn0search9turn2search1turn2search4  
- **Exactly‑once vs at‑least‑once:** нормативно фиксировать выбор: default stream проще и соответствует at‑least‑once; если нужны exactly‑once гарантии — использовать committed streams и offsets, что повышает сложность. citeturn2search1turn2search4

Go‑интеграция:

- Для query‑контурa использовать официальный Go client library (`cloud.google.com/go/bigquery`). citeturn0search29  
- Для стриминга через Storage Write API в Go есть клиентский пакет `cloud.google.com/go/bigquery/storage/managedwriter`, который «оборачивает» BigQuery Write API client. citeturn2search16turn2search1

### Дефолты для Druid‑подобных систем на примере Apache Druid

entity["organization","Apache Software Foundation","open-source foundation"] поддерживает Apache Druid — аналитическое хранилище, где данные организованы по времени в **неизменяемые сегменты**. Druid прямо формулирует: данные хранятся в immutable files (segments), а операции data management включают замену/удаление сегментов, schema changes и compaction (переиндексацию существующих данных ради footprint/performance). citeturn10search8turn10search3turn9search3

Нормативные дефолты ingestion/layout:

- **Time chunk partitioning:** Druid описывает time chunk partitioning и рекомендует практические варианты granularity; для streaming ingestion часто выбирают `hour`, чтобы compaction мог следовать ближе к ingestion по времени. citeturn7search0turn1search3  
- **Secondary partitioning:** Druid дополнительно делит time chunk на immutable segments; вторичное партиционирование по «естественному» измерению улучшает locality, компрессию и производительность запросов. citeturn7search0turn10search0  
- **Rollup:** Druid поддерживает rollup (предагрегацию) при ingestion, что может «на порядки» уменьшить размер/row count, но за счёт потери возможности query individual events. Это должно быть оформлено как явный trade‑off в стандарте. citeturn2search3turn2search0  
- **Compaction:** Druid подчёркивает, что query performance зависит от оптимального размера сегментов; compaction читает существующие сегменты за интервал, комбинирует и создаёт новый набор сегментов. citeturn9search3turn0search2  
- **Late‑arriving data:** для streaming ingestion Druid предоставляет механизмы ограничения «поздних сообщений» через `lateMessageRejection*`, чтобы предотвращать concurrency issues при нескольких пайплайнах (например streaming + nightly batch). citeturn8view0turn7search2  
- **Exactly‑once ingestion:** для streaming ingestion Druid заявляет «exactly‑once stream processing guarantee»; Kafka ingestion описывает exactly‑once через offsets/partition mechanism, а Supervisor APIs поясняют, что indexing service хранит latest persisted offsets для exactly‑once across tasks. citeturn17search5turn17search0turn17search16  
- **Retention:** retention реализуется правилами load/drop; Druid предупреждает, что для retention «retain only data from a defined period» нужно обязательно задавать и drop rule, иначе будет действовать default `loadForever`. citeturn3search0turn3search4  
- **Обновления/перезагрузка:** модель immutable segments означает, что «обновление данных» обычно делается через reingest/overwrite по time chunks/segments (batch ingestion overwrite mode, иногда с предварительным drop). citeturn10search8turn10search1turn10search4

## Матрица решений и компромиссы

Ниже — практическая «матрица» в виде правил выбора, ориентированная на predictable production‑поведение и минимизацию hallucinations при генерации кода/DDL.

### Когда колонночная аналитическая БД оправдана

Выбирайте аналитическое хранилище (ClickHouse/BigQuery/Druid‑класс), если выполнены одновременно условия:

- Основной запросный профиль — сканирование и агрегации по большим объёмам (а не точечные чтения/апдейты), и вы готовы поддерживать partitioning/кластеризацию/rollups/MVs ради стоимости и latency. Это соответствует свойствам колонночных систем и практикам оптимизации (partition pruning, block pruning, pre‑aggregation). citeturn0search19turn0search1turn15view0turn2search3turn1search0  
- Вам подходит «asynchronous ingestion» как норма (микробатчи/стрим), а корректность на уровне отдельных событий допускает задержки обработки/уплотнения (merge/compaction/refresh). ClickHouse описывает async inserts и фоновые merge, Druid — compaction и immutability сегментов, BigQuery — materialized views refresh‑поведение и write API semantics. citeturn4view0turn9search3turn13search3turn2search1  
- У вас есть чёткая стратегия retention и управления стоимостью: TTL/partition expiration/drop rules. Все три системы предоставляют «удаление по времени» на уровне партиций/сегментов (ClickHouse TTL + partitions, BigQuery partition expiration, Druid drop rules). citeturn0search4turn3search2turn3search0

### Когда выбор аналитической БД является ошибкой

Считайте выбор аналитической БД ошибочным (для основного контура микросервиса), если:

- Требуются частые row‑level `UPDATE/DELETE` и жёсткая транзакционность. ClickHouse маркирует обновления/удаления как «mutations» через `ALTER TABLE`; отдельные best practices требуют избегать частых/крупных мутаций. BigQuery указывает rate limits для DML и необходимость backoff при превышении скоростей, а также в материалах о мутациях прямо противопоставляет warehouse‑модель OLTP‑подходу по частоте мутаций. Druid, будучи сегмент‑ориентированным, оперирует заменой/удалением сегментов и reingest для обновлений. citeturn14search1turn9search14turn9search16turn10search8turn10search1  
- Нужна «точечная» низколатентная выдача по ключу как доминирующий кейс (например “GET by id” в p99) и вы рассчитываете на B‑tree‑подобные уникальные индексы. ClickHouse прямо объясняет, что его primary index — sparse (работает по гранулам), primary key не уникален и оптимизирован для skipping ranges, а не для OLTP‑style single‑row lookups. citeturn12view1turn18search14turn18search1  
- Команда не готова к «дисциплине» partitioning/filters. В BigQuery отсутствие корректного partition filter может привести к сканированию всех партиций и росту стоимости/latency; BigQuery даёт явные best practices и даже позволяет обязать фильтр. В Druid отсутствие корректной стратегии segmentGranularity/compaction приводит к неоптимальным сегментам (официально: производительность зависит от оптимального размера сегментов, compaction оптимизирует). В ClickHouse неправильный выбор partition key/ORDER BY может ухудшить запросы и усложнить life‑cycle операции. citeturn13search4turn9search3turn11view0turn12view1

### Trade-offs между ClickHouse, BigQuery и Druid‑классом

- **Операционные затраты vs managed‑сервис.** BigQuery снимает большую часть ops‑нагрузки, но стоимость запросов привязана к processed bytes, и это накладывает дисциплину на partitioning/clustering и на сами запросы (включая то, что `LIMIT` не снижает bytes processed). ClickHouse/Druid чаще требуют кластерной эксплуатации, но дают предсказуемые latency‑паттерны при корректном дизайне схем/сегментов/compaction. citeturn0search5turn9search3turn17search15  
- **Pre-aggregation подход.** ClickHouse тащит предагрегации через materialized views/движки агрегации на insert time; Druid — через rollup (с явной потерей event‑level запросов); BigQuery — через materialized views и (часто) через дисциплину partition/clustering + MV ограничения/refresh. citeturn1search0turn2search3turn13search0turn13search3  
- **Семантика «обновлений».** ClickHouse предлагает «эмуляцию апдейта» engine‑ами (`ReplacingMergeTree`) с фоновыми merge и без обещания немедленного отсутствия дублей; Druid реализует обновления через замену сегментов/reingest; BigQuery поддерживает DML, но мутации подчинены rate limits/очередям jobs и чаще хуже подходят для OLTP‑частоты. citeturn1search1turn10search8turn9search14turn9search16

## Правила для LLM

Ниже — набор правил, который стоит положить в LLM‑instruction docs, чтобы модель генерировала предсказуемый Go‑код/DDL/queries для аналитических БД без «магии». Формат: MUST / SHOULD / NEVER.

### MUST

LLM MUST:

1) **Указывать и документировать выбранную семантику доставки данных (at‑least‑once vs exactly‑once) и соответствующий механизм.**  
   - Для BigQuery: default stream = at‑least‑once; для exactly‑once нужны application‑created streams и offsets. citeturn2search1turn2search4  
   - Для Druid streaming ingestion декларируется exactly‑once, реализуемое через offsets и supervisored ingestion. citeturn17search5turn17search16  
   - Для ClickHouse: идемпотентные ретраи основаны на «консистентных батчах» и дедупликации вставок; при необходимости использовать `insert_deduplication_token`. citeturn4view0turn3search1turn3search13

2) **Проектировать ingestion как batch‑friendly.**  
   - Для ClickHouse: батчи минимум ~1000 строк, желательно 10k–100k; при невозможности — async inserts. citeturn4view0turn2search17  
   - Для Druid: ingestion оформлять через спецификации/задачи, с заданными granularity и partitioning. citeturn2search6turn7search0

3) **Делать time‑partitioning стратегию явной и согласованной с retention.**  
   - ClickHouse: выравнивать `PARTITION BY` по тому же времени, что и TTL (drop partitions эффективнее row‑delete). citeturn0search4turn11view0  
   - BigQuery: использовать partition expiration и (по умолчанию) `require_partition_filter`. citeturn3search2turn6view0turn13search4  
   - Druid: retention через load/drop rules. citeturn3search0turn3search4

4) **Генерировать запросы, совместимые с pruning (а не «красивые», но дорогие).**  
   - BigQuery partition pruning требует константных/простых фильтров и изоляции partition column; clustering требует простых фильтров по clustered columns без функций/сравнения колонка‑к‑колонке. citeturn13search4turn15view0  
   - ClickHouse должен фильтровать по primary key/ORDER BY префиксу, чтобы sparse index мог пропускать гранулы; primary key не уникален и работает батчево по гранулам, поэтому запросы должны учитывать природу range scan. citeturn12view1turn18search1turn18search3  
   - Druid партиционирован по времени и сегментам; запросы должны иметь time bounds и соответствовать time‑chunk модели. citeturn10search8turn7search0

5) **В Go всегда использовать `context.Context` для запросов/вставок и задавать таймауты.** Go‑документация описывает контекст как стандартный механизм отмены in‑progress операций; `database/sql` поддерживает контекст в `BeginTx` и при отмене контекста может rollback’ать транзакцию. citeturn13search2turn13search6

6) **Параметризовать запросы и запрещать конкатенацию SQL из пользовательского ввода.** OWASP закрепляет parameterized queries как базовую защиту от SQL Injection. citeturn7search3turn7search5

### SHOULD

LLM SHOULD:

1) **Ставить pre‑aggregation по умолчанию там, где это уместно, но фиксировать потери/ограничения.**  
   - ClickHouse: materialized views «переносят вычисления на insert time» и инкрементально обновляются при вставках. citeturn1search0turn1search4  
   - Druid: rollup может «на порядки» уменьшить размер, но убирает возможность query individual events — это должно быть явно отражено в ADR/документации. citeturn2search3turn2search0  
   - BigQuery: materialized view data нельзя модифицировать напрямую DML/LOAD и существуют ограничения/квоты; при невозможности инкрементального обновления cached data может не использоваться до refresh. citeturn13search0turn13search3turn13search9

2) **Проектировать late‑arriving data как стандартный кейс.**  
   - Druid: имеет механизмы `lateMessageRejection*` и настройки auto‑compaction с учётом late data (`skipOffsetFromLatest`). citeturn8view0turn9search0  
   - ClickHouse: для «апдейтов/поздних коррекций» предпочитать append‑only версии событий + движки/стратегии дедупликации, понимая асинхронность merge. citeturn1search5turn1search1turn3search5  
   - BigQuery: выбирать write‑механизм (at‑least‑once vs exactly‑once) согласно бизнес‑цене дублей и сложности offsets/streams. citeturn2search1turn2search4

3) **Оформлять спорные оптимизации как опциональные и сопровождать trade‑offs.** Например:  
   - ClickHouse `OPTIMIZE … FINAL` — административный инструмент, но дорогой для ежедневной эксплуатации. citeturn14search9turn14search0  
   - BigQuery: sharding таблицами по датам — анти‑паттерн, лучше time‑partitioned. citeturn13search8

### NEVER

LLM NEVER:

1) **NEVER предполагать, что primary key в ClickHouse уникален.** Документация ClickHouse прямо говорит, что primary key не уникален и определяет порядок сортировки/модель sparse index. citeturn18search14turn12view1

2) **NEVER добавлять `OPTIMIZE … FINAL` в request path или cron «на всякий случай».** ClickHouse описывает `OPTIMIZE FINAL` как дорогую, CPU/I/O‑интенсивную операцию и рекомендует избегать как повседневной практики. citeturn14search0turn14search9

3) **NEVER генерировать BigQuery запросы без partition filter к большим таблицам, если таблица partitioned.** BigQuery прямо указывает, что без фильтра может не быть partition elimination; также есть опция «require partition filter» и ошибка при попытке сканировать без фильтра. citeturn13search4turn6view0

4) **NEVER использовать clustered columns в BigQuery внутри функций/сложных выражений в фильтре, если ожидаете block pruning.** BigQuery прямо говорит, что pruning не применяется при complex filter expressions (например `CAST(cluster_col AS STRING) = ...`). citeturn15view0

5) **NEVER проектировать Druid как систему «row‑level updates».** Druid хранит данные в immutable segments; обновления/удаления/замены выполняются на уровне сегментов/интервалов через data management workflows (replace/delete/reindex/compaction). citeturn10search8turn10search3turn10search1

## Примеры и анти‑паттерны

### Пример: ClickHouse ingestion в Go с батчингом, идемпотентностью и контекстом

**Хорошо (скелет):** отдельный ingestion‑компонент, батчинг, таймаут, «консистентный батч» + токен для дедупликации на ретраях.

```go
// Пример-скелет: батчинг insert'ов в ClickHouse.
// В реальном сервисе добавьте метрики, ограничение памяти, backpressure, DLQ.
package ingest

import (
	"context"
	"crypto/sha256"
	"encoding/hex"
	"time"

	"github.com/ClickHouse/clickhouse-go/v2"
)

type Event struct {
	EventTime time.Time
	TenantID  string
	Name      string
	Value     float64
	EventID   string // стабильный id события
}

type Writer struct {
	conn clickhouse.Conn
}

func NewWriter(conn clickhouse.Conn) *Writer { return &Writer{conn: conn} }

func (w *Writer) InsertBatch(ctx context.Context, events []Event) error {
	// Таймаут на батч: не даём insert'у висеть бесконечно.
	ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
	defer cancel()

	// Идемпотентный токен: зависит от содержимого батча и порядка.
	// ClickHouse рекомендует консистентные батчи для идемпотентных ретраев.
	h := sha256.New()
	for _, e := range events {
		h.Write([]byte(e.EventID))
		h.Write([]byte{0})
	}
	dedupToken := hex.EncodeToString(h.Sum(nil))

	batch, err := w.conn.PrepareBatch(ctx,
		`INSERT INTO events (event_time, tenant_id, name, value, event_id)
		 VALUES (?, ?, ?, ?, ?)`,
		clickhouse.WithSettings(clickhouse.Settings{
			"insert_deduplication_token": dedupToken,
		}),
	)
	if err != nil {
		return err
	}

	for _, e := range events {
		if err := batch.Append(e.EventTime, e.TenantID, e.Name, e.Value, e.EventID); err != nil {
			return err
		}
	}
	return batch.Send()
}
```

Обоснование: ClickHouse рекомендует крупные батчи и консистентные батчи для идемпотентных ретраев, а также предоставляет `insert_deduplication_token` для управления дедупликацией. Контекст и таймауты в Go — стандартный способ управления in‑progress операциями. citeturn4view0turn3search13turn3search1turn13search2

**Плохо (анти‑паттерн):** один insert на событие в request path + `OPTIMIZE FINAL` «чтобы не было дублей».

```go
// ПЛОХО: не делайте так
db.Exec("INSERT INTO events VALUES (...)")
db.Exec("OPTIMIZE TABLE events FINAL")
```

ClickHouse отдельно объясняет, что `OPTIMIZE … FINAL` дорог, а мелкие частые вставки порождают множество parts и лишнюю работу; вместо этого рекомендуется батчить или использовать async inserts. citeturn14search0turn4view0turn16view0

### Пример: BigQuery запросы, совместимые с pruning

**Хорошо:** partition filter константный/изолированный; clustering columns — без функций.

```sql
-- Хорошо: partition pruning + block pruning (если CLUSTER BY по customer_id).
SELECT
  customer_id,
  COUNT(*) AS events
FROM dataset.events
WHERE event_date BETWEEN DATE '2026-02-01' AND DATE '2026-02-28'
  AND customer_id = 10000
GROUP BY customer_id;
```

BigQuery даёт явные best practices для partition pruning (константные выражения, изоляция partition column) и для clustered tables (не использовать функции над clustered columns). citeturn13search4turn15view0

**Плохо:** «красивый» фильтр, который ломает pruning.

```sql
-- Плохо: вычисление даты через выражение может заставить сканировать все партиции.
WHERE DATE(TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)) <= event_date

-- Плохо: функция над clustered column ломает block pruning.
WHERE CAST(customer_id AS STRING) = "10000"
```

BigQuery прямо предупреждает про динамические/сложные выражения для partition pruning и про complex filter expressions для clustering. citeturn13search4turn15view0

### Пример: Druid — rollup, late data, retention

**Хорошо (концептуально):** разделить «raw datasource» и «rollup datasource» (или хотя бы документировать потерю event‑level), выбирать `segmentGranularity` осознанно и включать compaction/retention правила.

- Rollup уменьшает хранимый объём, но теряет возможность query individual events — значит, rollup должен быть либо вторым слоем (витрина), либо конечной формой данных только если бизнес не требует event‑level. citeturn2search3turn2search0  
- Для streaming ingestion учитывать late messages через `lateMessageRejectionPeriod`/`lateMessageRejectionStartDateTime`, а также планировать auto‑compaction с `skipOffsetFromLatest` в зависимости от частоты late data. citeturn8view0turn9search0  
- Retention обязательно оформлять drop rules + load rules, иначе данные останутся под `loadForever`. citeturn3search0turn3search8  
- Обновления — через replace/reingest сегментов/интервалов, а не «UPDATE row». citeturn10search8turn10search4turn10search1

### Типичные ошибки и LLM‑галлюцинации

1) **«Primary key = уникальность» в ClickHouse.** Это распространённая переносная привычка из OLTP; в ClickHouse primary key определяет порядок хранения и sparse index, но уникальность не обеспечивается. citeturn18search14turn12view1

2) **«Partitioning — главная оптимизация запросов» в ClickHouse.** ClickHouse подчёркивает, что partitioning в первую очередь data management, и может ухудшить запросы, если запросы ходят по многим партициям/parts. citeturn11view0turn12view1

3) **Использование `ReplacingMergeTree` как «настоящего upsert» без понимания merge.** Документация: дедупликация происходит только во время merge, merge асинхронен и отсутствие дублей не гарантировано; `OPTIMIZE` нельзя использовать как планируемый механизм, он может читать/писать большой объём данных. citeturn1search1turn14search5

4) **BigQuery: ожидание, что `LIMIT` уменьшит стоимость.** BigQuery pricing прямо говорит обратное: вы платите за обработанные данные в выбранных столбцах даже при `LIMIT`. citeturn0search5

5) **BigQuery: «достаточно партиционировать» без `require_partition_filter` и без дисциплины фильтров.** BigQuery показывает, что динамические/сложные выражения могут ломать pruning, и даёт механизм, который запрещает запросы без фильтра. citeturn13search4turn6view0

6) **Druid: попытка «исправлять строки» и ожидание row‑level изменений.** Druid хранит immutable segments; data management — это операции замены/удаления сегментов, compaction и reindex. citeturn10search8turn10search3turn10search1

## Чек‑лист ревью и структура файлов

### Review checklist для PR/code review

Этот чек‑лист предназначен для PR’ов, где добавляется/меняется аналитическое хранилище, схемы, ingestion и запросы.

**Архитектура и корректность доставки**
- Зафиксировано, какая семантика нужна: at‑least‑once или exactly‑once; выбранный механизм соответствует системе (BigQuery default stream vs application‑created streams/offsets; Druid streaming ingestion offsets; ClickHouse консистентные батчи/токены). citeturn2search1turn2search4turn17search16turn4view0turn3search13  
- Ingestion вынесен из request path или имеет явные ограничения/очереди/таймауты (нет «вставки по одной строке на HTTP‑запрос» по умолчанию). citeturn4view0turn16view0turn2search9  
- Late‑arriving data обработаны: есть стратегия «окна поздних данных», backfill, и понимание, как это влияет на rollups/MVs. citeturn8view0turn9search0turn1search0

**Схема и физический layout**
- ClickHouse: `PARTITION BY` выбран как data management (retention), а не случайно; `ORDER BY`/primary key соответствует типовым фильтрам, и в PR явно перечислены «главные запросы», под которые выбирался ключ. citeturn11view0turn18search0turn12view1  
- ClickHouse: нет частых мутаций и нет `OPTIMIZE … FINAL` как регулярной практики; если есть — в PR есть объяснение и оценка влияния. citeturn14search1turn14search0turn14search9  
- BigQuery: partitioned tables создаются с `require_partition_filter = TRUE` (если это не ломает продуктовый сценарий) и с осознанным распределением `PARTITION BY`/`CLUSTER BY`. citeturn6view0turn1search2  
- Druid: `segmentGranularity` и secondary partitioning обоснованы; rollup включён только тогда, когда потеря event‑level явно приемлема; compaction/retention настроены. citeturn7search0turn2search3turn9search3turn3search0

**Запросы, стоимость, производительность**
- BigQuery: запросы совместимы с partition pruning и block pruning (не оборачивают partition/cluster columns в функции и не используют сравнения колонка‑к‑колонке в pruning‑критичных местах). citeturn13search4turn15view0  
- BigQuery: нет ложного предположения, что `LIMIT` уменьшает стоимость — проверены `SELECT`‑списки и ожидаемый bytes processed. citeturn0search5  
- ClickHouse: запросы используют фильтры по ключам так, чтобы sparse index реально работал (понимание granules/marks); не добавлен `FINAL`/`OPTIMIZE` как «магическая кнопка». citeturn12view1turn14search3turn14search0  
- Druid: запросы ограничены по времени (time bounds), соответствуют time‑chunk модели, и не заставляют сканировать «вечность». citeturn10search8turn7search0  

**Go‑реализация и безопасность**
- Везде используются `context`‑варианты вызовов и таймауты; нет «вечных» запросов без отмены. citeturn13search2turn13search6  
- SQL/запросы параметризованы; нет конкатенации строк с пользовательским вводом (OWASP). citeturn7search3turn7search5
