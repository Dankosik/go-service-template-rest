# Область применения

Подход годится для greenfield Go‑микросервисов, где требуется зрелый, надежный REST API с документацией (например, OpenAPI), поддержкой версий и широким набором функциональности (фильтрация, пагинация, частйные обновления и т.д.). Это **не** канонический гайд для каждого простого сервиса (например, небольших CRUD-сервисов без сложных операций). Если нужны очень специфические или узконаправленные API (например, графовые запросы или чисто внутренние пиринги), некоторые рекомендации можно ослабить. Тем не менее, даже во внутренних сервисах полезны стандарты именования, обработки ошибок и явного версионирования. 

# Рекомендуемые настройки по умолчанию

- **Версионирование**: включать _только номер мажорной версии_ в путь URI как первый сегмент, например `/v1/…`【27†L132-L139】. Это наиболее распространенный и простой метод (по аналогии с Google AIP)【27†L132-L139】. Альтернативой является версионирование через заголовки или media-type (например, `Accept: application/vnd.myapi.v1+json`), но оно сложнее для разработки и отладки. (Часть источников рекомендуют media-type вместо URL【38†L139-L144】, однако для зеленфилд-проекта стандартный подход с URI версии будет более привычен.)

- **Именование ресурсов**: **существительные в URI, предпочтительно во множественном числе**. Например, `/orders`, `/customers/{id}/orders`【5†L168-L176】【32†L150-L157】. HTTP-глаголы в пути не нужны (например, `/get-order` – *неправильно*)【5†L168-L176】【32†L132-L140】. Именование сегментов пути должно отражать бизнес-сущности, а не внутреннюю структуру БД【5†L220-L228】. Коллекция ресурсов (например, «заказы») именуется во множественном (`orders`), а отдельный ресурс имеет составной URI `/orders/{orderID}`【5†L172-L180】【5†L243-L252】. Идентификаторы ресурсов (ID) обычно — короткие строки или числа, строчные, без спецсимволов (RFC-1123/RFC-1034)【16†L167-L175】【16†L250-L258】.  

- **Связи между ресурсами**: допускается иерархия URI (`/customers/{id}/orders`), но не нужно создавать чрезмерно глубокие пути (`/a/{}/b/{}/c/{}`). Предпочтительнее давать простые коллекции и переходить к ним по идентификатору. Например, чтобы найти продукты заказа, можно сначала получить `/orders/{id}` и из ответа взять ссылку или ID, затем вызвать `/orders/{id}/products` или `/products?orderId={id}`【5†L181-L190】【5†L195-L201】. Не стоит зеркалить структуру БД или делать «чати́» API (много мелких ресурсов)【5†L208-L217】【5†L220-L228】.

- **Методы HTTP**: использовать стандартные глаголы как предписано:
  - `GET` – получить ресурс или коллекцию, идемпотентно.
  - `POST` – создать ресурс в коллекции (URI коллекции), сервер присваивает новый ID【22†L310-L318】.
  - `PUT` – полное обновление или создание ресурса по заданному URI, идемпотентно (несколько повторов не изменят итог)【22†L331-L339】【22†L344-L352】.
  - `PATCH` – частичное обновление ресурса по URI, не обязательно идемпотентно (может применять только указанные изменения)【22†L369-L378】【22†L428-L436】.
  - `DELETE` – удаление ресурса по URI【7†L437-L446】.
  
  Четко разделять операции: например, не использовать `GET` для изменения состояния или `POST` для чтения. Массивные обновления могут выполняться через `PUT` или отдельные «batch» API (см. ниже)【22†L344-L352】【36†L539-L548】.  

- **Коды статуса**: возвращать подходящие HTTP-коды:
  - Успех: `200 OK` (GET/PUT/PATCH успех, тело с ресурсом), `201 Created` (успешно создан ресурс, Location с новым URI)【22†L314-L318】【22†L353-L360】, `204 No Content` (успех без тела, например DELETE или PUT с пустым ответом)【22†L353-L360】【7†L430-L436】.
  - Клиентские ошибки: `400 Bad Request` (неверный запрос, неверный формат)【22†L319-L324】, `404 Not Found` (ресурс отсутствует)【7†L437-L446】, `409 Conflict` (конфликт версий или нарушение ограничений)【22†L353-L360】【7†L430-L434】.
  - Аутентификация/авторизация: `401 Unauthorized`, `403 Forbidden` (по ситуации).
  - **Идемпотентность**: PUT, GET, DELETE считаются идемпотентными (повтор - без изменения), POST/PATCH – не гарантированно【22†L344-L349】. При реализации следует поддерживать идемпотентность PUT (что клиентский запрос повторно приводит к тому же состоянию ресурса).

- **Параметры запросов**:
  - **Пагинация**: поддерживать разбиение больших коллекций. Часто используют параметры `limit` (размер страницы) и `offset` (смещение)【36†L530-L539】【36†L541-L549】. Установить разумные дефолты (напр. `limit=25, offset=0`) и максимум (чтобы избежать DoS)【36†L541-L549】. В продвинутых случаях можно предусмотреть cursor‐based пагинацию (см. Zalando【38†L129-L134】) или возвращать «next page token». Но по умолчанию OFFSET-пагинация проста. Ответ может включать метаданные страниц (total, nextLink) или заголовки.
  - **Фильтрация и сортировка**: принимать фильтры через query-параметры (напр. `?status=shipped&minCost=100`)【36†L548-L556】. Сортировку – параметром `sort=field` (или `sort=-field` для обратного)【24†L557-L565】. Замечание: каждый уникальный запрос по параметрам является разным URI, это может влиять на кеширование【24†L561-L570】. 
  - **Выбор полей (partial response)**: можно поддержать `fields` или `select`, чтобы отдавать только нужные поля, улучшая производительность. Например: `/orders?fields=id,amount`【24†L567-L574】. Проверять, что запрашиваемые поля доступны и безопасны. Кроме того, для очень больших бинарных ресурсов следует поддержать HTTP Range (Accept-Ranges) и HEAD для дозагрузки чанков【24†L578-L588】.
  
- **Ошибки и формат ошибок**: возвращать единообразную JSON-структуру ошибки. Например, как рекомендует Google AIP, `{ "error": { "code": <статус>, "message": "<описание>", "details": {…} } }`【42†L147-L155】. Сообщение должно быть понятным техническому пользователю и не содержать служебной информации【42†L152-L160】. Использовать «канонические» HTTP-коды для ошибок (см. [142–154]). Не следует возвращать 200 с телом ошибки. Также можно придерживаться стандарта RFC 7807 (problem details) или Google RPC Status.
  
- **Типы данных**: по умолчанию использовать JSON (`Content-Type: application/json; charset=utf-8`)【32†L72-L80】【22†L310-L318】. Другие форматы (XML, multipart, стриминг и т.д.) подключать только по необходимости (и документировать). Код должен обрабатывать и возвращать заголовок `Accept: application/json`.  
  
- **Длинные операции**: если обработка запроса занимает много времени, следует делать асинхронный шаблон. Тогда метод сразу отвечает `202 Accepted` и в заголовок `Location` кладёт URL для опроса статуса задачи【36†L485-L494】. Клиент потом по этому URL может проверить прогресс. Когда задача завершена и, например, создала ресурс, возвращать `303 See Other` с `Location` на новый ресурс【36†L513-L521】.  

- **Кеширование и ETag**: при GET отдавать заголовок `ETag`, отражающий версию ресурса (например, hash или счетчик)【29†L463-L472】. На последующих GET клиент передаёт `If-None-Match`; если ETag совпадает, сервер отвечает `304 Not Modified` (пустое тело)【29†L523-L532】【29†L533-L541】. Для обновлений (PUT/PATCH) поддерживать ETag+If-Match: клиент включает `If-Match` со старым ETag, сервер сравнивает, и если не совпадает, возвращает `412 Precondition Failed` (ресурс изменился за это время)【31†L661-L670】【31†L684-L693】. Если совпадает, выполняем обновление и можем вернуть 204 или 200. Это обеспечивает опережающую (optimistic) конкуренцию【31†L669-L678】【31†L684-L693】. (Без If-Match сервер может просто перезаписать ресурс, риск «затирания» изменений других пользователей.)

# Матрица решений и компромиссы

- **URL vs header versioning**: версия в пути проще реализовать и понять (любой клиент сразу видит версию в URI)【27†L132-L139】. Header-версионирование (через `Accept` или кастомный заголовок) считается «более чистым REST» и позволяет использовать один URL для всех версий, но затрудняет кеширование и отладку (клиентам нужно указывать заголовки)【38†L139-L144】. По умолчанию взять версию в пути; упомянуть header-альтернативу как **опциональную**.

- **Offest-пагинация vs курсорная**: OFFSET простая и интуитивная, но при большом смещении может быть медленной (каждый запрос сдвигает результаты). Курсорная (с `nextCursor`) требует хранить состояние или токен, но обеспечивает постоянную производительность. Для начала достаточно OFFSET (особенно если сервис невелик). Для высоконагруженных сценариев можно позже мигрировать на курсоры (Zalando рекомендует курсорную【38†L129-L134】).

- **PUT vs PATCH**: `PUT` требует полной модели ресурса, `PATCH` — часть полей. `PATCH` удобнее для клиентского кода (можно обновлять одно поле), но сложнее реализовать (сервер должен применять diff). Если ресурсы небольшие, можно поддерживать только `PUT`. Но лучше реализовать `PATCH` (например, JSON Merge Patch или JSON Patch) для гибкости【7†L369-L378】【7†L428-L436】. LLM-инструмент может делать любой из них; **решение**: в шаблоне поддерживать оба, но по умолчанию использовать JSON Merge Patch с `Content-Type: application/merge-patch+json`【7†L406-L415】.

- **Структура ошибок**: можно использовать Google RPC Status или Problem Details (RFC 7807). Если цель – интеграция с gRPC/Google экосистемой, использовать format AIP-193【42†L147-L155】. Если нет, простая схема `{error:{code,message,details}}` подходит. Важно: **консистентность**.  

- **Соглашения по API-first и OpenAPI**: для продакшн-сервиса **рекомендуется** иметь спецификацию OpenAPI/Swagger (в шапке `docs/` или корне репозитория). Это позволит генерировать SDK, документацию, тесты. 

# Правила (MUST/SHOULD/NEVER) для LLM

- **MUST** использовать в путях **существительные во множественном числе** для коллекций и идентификаторы для элементов (напр. `GET /orders`, `GET /orders/{id}`)【5†L168-L176】.  
- **MUST NOT** включать **глаголы** или операции в URI (не делать `/orders/create` или `/orders/delete`)【5†L168-L176】【32†L132-L140】.  
- **MUST** соответствовать протоколу HTTP: `GET` – чтение, `POST` – создание (в URI коллекции, без body ID)【22†L310-L318】, `PUT` – обновление (или создание по заданному URI, идемпотентно)【22†L331-L340】【22†L344-L352】, `DELETE` – удаление. `PATCH` – частичное обновление【22†L369-L378】【7†L428-L436】.  
- **MUST** возвращать правильный HTTP статус-код: `200/201/204` при успехе, `4xx` при ошибке клиента, `5xx` при сбое сервера. В ответе на создание возвращать **201 Created** и header `Location` с URI нового ресурса【22†L314-L318】. При удалении – `204 No Content`【7†L439-L446】.  
- **MUST** устанавливать заголовки `Content-Type: application/json` и отдавать JSON по умолчанию【32†L72-L80】.  
- **MUST** включать **версию API** в базовый путь (например, `/v1/`)【27†L132-L139】. Не использовать версию API в имени ресурса или теле данных.  
- **MUST** поддерживать пагинацию и фильтрацию (напр., `?limit=&offset=`, `?field=value`) на GET-коллекциях【36†L530-L539】【36†L548-L556】.  
- **MUST** документировать API через OpenAPI/Swagger (идентично спецификациям Google, Zalando – в качестве «must publish OpenAPI»【38†L154-L161】).  
- **SHOULD** возвращать **ETag** на GET (и обрабатывать `If-None-Match`) для кеширования【29†L523-L532】.  
- **SHOULD** использовать `If-Match` с ETag для PUT/PATCH (оптимистичная блокировка)【31†L669-L678】【40†L7120-L7129】.  
- **SHOULD** поддерживать булковые операции, если они часто требуются. Например, можно реализовать endpoint типа `PUT /orders` с body `{orders: [...]}` для обновления нескольких элементов【22†L344-L352】 или гибридный вариант BatchGet【18†L144-L152】. Но это **дополнительно**, не обязательно для первого шаблона.  
- **SHOULD** возвращать структурированные ошибки. Поле `error.message` – понятное описание, не бросайте в тело трассировку. Код ошибок – понятные и предсказуемые (например, `error.code = 404` для "не найдено"). Следуйте рекомендациям по стандартным кодам【42†L147-L155】.  
- **NEVER** раскрывать в сообщении ошибки внутренние детали сервера (стек, SQL-запросы).  
- **NEVER** принимать «твист» правил (например, менять метод HTTP без явной причины).  
- **NEVER** использовать нестабильные приемы: все изменения API должны быть обратно-совместимыми **по умолчанию** (MUST не ломать совместимость без версии)【38†L139-L144】. При внесении несовместимых изменений – повышать номер мажорной версии.  
- **MAY** рассмотреть использование заголовка `Idempotency-Key` для POST (как советует Zalando)【39†L46-L49】, особенно если клиенты могут повторять запросы (такая практика есть у Stripe и Google).  

# Примеры

**Хороший пример маршрутов**:
```go
// GET список заказов с фильтрацией и пагинацией
router.GET("/v1/orders", getOrders)            // /orders?status=shipped&limit=20&offset=40
// GET один заказ по ID
router.GET("/v1/orders/:orderId", getOrderByID)
// POST создание заказа в коллекции
router.POST("/v1/orders", createOrder)         // тело JSON без ID
// PUT полное обновление или создание заказа по заданному ID
router.PUT("/v1/orders/:orderId", updateOrder)
// PATCH частичное обновление заказа
router.PATCH("/v1/orders/:orderId", patchOrder)
// DELETE удаление заказа
router.DELETE("/v1/orders/:orderId", deleteOrder)
```
Пример **возвращаемого JSON** (GET /orders/123):
```json
{
  "id": "123",
  "status": "shipped",
  "amount": 49.99,
  "customerId": "77",
  "items": [ ... ]
}
```
Пример **ответа ошибки** (404 Not Found):
```json
{
  "error": {
    "code": 404,
    "message": "Order 123 not found",
    "details": {"requestedId": "123"}
  }
}
```
Соответствующие заголовки: `Content-Type: application/json`, HTTP-код 404. 

**Плохие примеры**:
- `GET /v1/getOrder?id=123`: содержит глагол и query вместо чистого пути.  
- `POST /v1/orders/123`: попытка создать ресурс по заранее известному ID – должен быть либо `PUT`, либо сервер бросит 400.  
- `GET /v1/orders` **без** пагинации по умолчанию, возвращающий всё сразу – может навредить производительности.  
- `PUT /v1/orders` для **создания** заказа без ID: неясно, какой ID присвоен; лучше использовать `POST /orders`.  
- Отправка ошибок как `200 OK` с `{success:false}` в теле – нарушает идеометику HTTP и усложняет клиентам нормальную обработку кода ответа. 

# Антипаттерны и типичные ошибки LLM

- **Дублирование БД-структуры**: например, `GET /v1/tables/user` – отражает внутреннее строение. REST требует бизнес-сущности, а не точную таблицу.  
- **Глубокие вложенные пути**: `GET /companies/1/departments/2/employees/3/addresses/4` – слишком сложно. Лучше запрашивать сначала компанию, затем по отдельности нужные ресурсы.  
- **Избыточные ресурсы (чати́ API)**: множество мелких эндпоинтов для каждой связи, вместо использования связей в теле ответа.  
- **Непоследовательное использование статусов**: пример: при «не найдено» возвращают 200 с пустым объектом или строковым сообщением – клиенты не поймут, было ли это успешным результатом.  
- **Отсутствие валидации и явной сериализации**: LLM может сгенерировать код без проверки входа (например, `id := c.Param("id")` без проверки), или без явной привязки JSON и проверки ошибок. Следует убедиться, что код валидирует поля (например, непустые обязательные поля в POST/PATCH) и возвращает `400 Bad Request` при ошибке формата/валидации.  
- **Непредсказуемая сериализация**: например, использование `omitempty` на полях, из-за чего клиент может получить переменный список полей. Лучше явно форматировать JSON ответов.  
- **Игнорирование контекста**: забыть обрабатывать `context.Context` у запроса (таймауты и отмены), хотя для высоконагруженного сервиса важно корректно прекращать операции.  

# Чек-лист для PR/Code Review

- **URI и методы**: все эндпоинты используют существительные, методы HTTP соответствуют действиям? Примеры: `/v1/users` (COLLECTION), `/v1/users/{id}` (ITEM). Нет ли «verb» в URI?  
- **Версия API**: присутствует ли префикс `/v1`?  
- **Типы запросов**: правильно ли используются GET/POST/PUT/PATCH/DELETE? Нет ли, например, `GET` с телом запроса?  
- **Коды ответов**: возвращаются ли корректные HTTP-коды (201 на создание, 204 на успешное удаление, 404 если нет ресурса, 400 на неверный ввод и т.д.)? Пример: при создании – действительно ли отправлен 201 и Location?  
- **Формат JSON**: ответы всегда в JSON? Content-Type задан? Ошибки возвращаются как JSON с полем `error` или аналогичным?  
- **Валидация**: при получении данных из клиента (JSON) проверяется ли корректность (наличие обязательных полей, типы)? При нарушении – отдаётся ли 400?  
- **Пагинация и фильтры**: на коллекциях поддерживаются limit/offset и базовая фильтрация? Не возвращается ли «все записи» без ограничений?  
- **Concurrency/ETag**: при обновлении ресурсов реализовано ли сравнение ETag/версии? Если используется If-Match, проверяется ли он и при несоответствии возвращается 412?  
- **Идемпотентность**: idempotency-ключи (если есть) корректно обрабатываются? Повтор одного и того же POST не приводит к дублирующемуся ресурсу?  
- **Безопасность**: нет ли утечек конфиденциальной информации через ошибки или логи? Все запросы на чувствительные действия проверяют авторизацию. (Это выходит за рамки REST, но упомянуть стоит.)  
- **Документация**: есть ли схема/спецификация (OpenAPI)? Все модели данных и коды ошибок документированы?  
- **Антипаттерны**: нет ли в коде hardcode путей, «магических строк» вместо констант HTTP-кодов, дублирования URL-частей, отсутствия пагинации, и т.д.?

# Файлы для шаблонного репозитория

Из полученного материала логично оформить **отдельные markdown-документы в папке `docs/`**, например:

- `docs/api_style.md` – рекомендации по REST (имена ресурсов, методы, коды, пагинация, фильтрация, версионирование и т.д.) (информация из разделов *Рекомендуемые настройки*, *Матрица решений*).  
- `docs/error_handling.md` – правила формата ошибок и статус-кодов (информация из разделов про **Ошибки и формат ошибок**, требования AIP-193【42†L147-L155】).  
- `docs/language_model_guidelines.md` – правила поведения LLM (в стиле MUST/SHOULD/NEVER, из соответствующего раздела).  
- `docs/examples.md` – хорошие и плохие примеры API и кода (части из *Примеры*). Здесь можно вставить сниппеты Go, JSON.  
- `docs/checklist.md` – чек-лист для ревью PR (короткий список вопросов).  
- `docs/partial_responses.md` – (опционально) детали о полных/частичных ответах (range-запросы, header HEAD, поля fields).  
- **Дополнительно**: файл спецификации API (например, `openapi.yaml`) в корне или `docs/`, с шаблонными ресурсами, чтобы LLM мог генерировать код по нему.

Все эти документы станут частью шаблонного репозитория, задавая стандартный стиль API и инструкции LLM для разработки production-ready Go-сервиса.

**Источники:** официальные рекомендации Microsoft, Google и Zalando【5†L168-L176】【22†L344-L349】【27†L132-L139】【36†L530-L539】【42†L147-L155】, а также общепризнанные best practices.
