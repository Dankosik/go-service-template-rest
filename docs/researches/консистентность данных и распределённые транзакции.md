# Engineering standard и LLM-инструкции для production-ready template микросервиса на Go

## Scope

Этот стандарт и template репозиторий рассчитаны на «boring, battle‑tested» greenfield микросервис на Go, который должен быть быстро подхвачен человеком и LLM‑инструментами без “догадок” о скрытых архитектурных решениях. Он оптимален, когда:

- сервис разворачивается как контейнер в orchestrated среде (чаще всего entity["organization","Kubernetes","container orchestration project"]), и требуется корректное поведение по health‑checks, graceful shutdown и наблюдаемость; readiness/liveness probes и lifecycle semantics — часть контрактов выполнения citeturn1search6turn1search2turn4search16turn4search4  
- нужен предсказуемый и идиоматичный Go‑код, соответствующий официальным рекомендациям (style, error handling, testing); это напрямую влияет на «понимаемость» кода LLM’ом и на стабильность ревью citeturn0search0turn0search1turn12search0turn7search1turn7search18  
- сервис должен быть безопасным “by default”: HTTPS/TLS, ограничение размеров/таймаутов, защита от типовых классов атак на API (BOLA/authorization, инъекции, SSRF), and basic supply‑chain hygiene citeturn4search3turn1search3turn9search1turn9search2turn0search9turn3search11turn15search0turn15search1  
- команда хочет унифицировать «как мы пишем сервисы» и «как LLM пишет код в наших сервисах» в виде формальных правил, PR‑чеклистов и репо‑конвенций.

Этот подход применять НЕ стоит или нужно ослабить/переосмыслить, если:

- это не микросервис, а библиотека с публичным API (там другая политика по совместимости, stabilizing packages, go directive и т.п.); важно учитывать, что `go` directive — строгий минимум toolchain’а начиная с Go 1.21, и выбор версии влияет на потребителей citeturn17view1turn17view2turn20search0  
- требуется крайне высокая производительность или очень низкие задержки, и вы готовы ради этого отходить от “boring defaults” (например, особые аллок‑профили, агрессивные custom pools, нетипичные сетевые стеки) — этот стандарт сознательно консервативен и “не выжимает 100%” ценой сложности  
- у вас специфическая платформа деплоя (не контейнеры, не оркестрация, иные требования к логам/конфигурации), и 12‑factor ориентиры (env config, stdout logs, disposability) не соответствуют среде citeturn18search0turn9search0turn18search1  
- требуется строгая распределённая транзакционность/глобальная сериализуемость между сервисами. В современных больших системах распределённые транзакции часто оказываются непрактичными из‑за хрупкости и стоимости (в т.ч. блокирующее поведение 2PC при отказах); стандарт ниже специально ориентируется на local transactions + eventual consistency citeturn19view0turn19view2turn6search17turn5search1  

## Recommended defaults для greenfield template

Ниже — «рекомендованный минимальный комплект» для репозитория, который можно клонировать и сразу разрабатывать production‑ready сервис (и руками, и при помощи LLM). Формулировки намеренно нормативные; спорные места помечены как trade‑off.

### Базовые техрешения и версия Go

- **Go версии**: ориентир на актуальный стабильный релиз Go на дату 2026‑03‑02 — это Go 1.26 (релиз февраль 2026). citeturn2search0turn2search4  
- **`go.mod`: фиксировать `go` directive явно** и осознанно. Начиная с Go 1.21 `go` directive — это строгий минимальный requirement toolchain’а (раньше был “advisory”). citeturn17view1turn17view2turn3search20  
- **Рекомендация для template** (boring default для сервиса, не библиотеки):  
  - `go 1.26.0` (или `go 1.26` если вы принципиально не фиксируете patch)  
  - `toolchain go1.26.0` (явно, чтобы “suggested toolchain” был зафиксирован и не зависел от implicit поведения). Семантика `toolchain` и auto‑selection описана в официальных docs. citeturn17view1turn17view2turn3search20  
- **Важно**: `go mod init` в Go 1.26 по умолчанию может создать `go.mod` с версией на один релиз ниже (`go 1.25.0`). Это — tooling‑изменение релиза; в template репозитории версию нужно проставлять осознанно. citeturn20search0  

### Layout репозитория и границы пакетов

- **Один модуль на репозиторий** — boring default. Go wiki прямо отмечает, что multiple modules в одном repo поддерживаются, но обычно создают больше ongoing‑работы. citeturn8search20turn17view0  
- **`internal/` как основной контейнер для “непубличных” пакетов**. Официальная рекомендация по layout прямо говорит: держать пакеты в `internal` «как можно больше», чтобы иметь свободу refactor без внешних потребителей. citeturn17view0  
- **Правило импорта для `internal`** реализовано go tool’ом: импорт разрешён только коду внутри дерева directory‑ancestor `internal`. Это документировано в релиз‑нотах Go 1.4. citeturn8search4turn17view0  
- **Минимально достаточный layout для микросервиса** (сильный boring default, friendly для LLM):  
  - `cmd/<service>/main.go` — точка входа (чтобы однозначно: один бинарь, один main)  
  - `internal/app/` — wiring/DI (инициализация зависимостей)  
  - `internal/httpapi/` (или `internal/grpcapi/`) — транспортный слой  
  - `internal/domain/` — бизнес‑логика и модели (минимально необходимые интерфейсы)  
  - `internal/storage/` — реализации доступа к данным  
  - `internal/observability/` — логирование/трейсинг/метрики  
  - `internal/config/` — загрузка и валидация конфига  
  Такой подход отражает официальный guidance “организации Go module” и идею, что `internal` позволяет безопасно держать большинство пакетов непубличными. citeturn17view0turn8search4  

### HTTP по умолчанию, таймауты и безопасность транспортного слоя

- **HTTP сервер по умолчанию**: стандартный `net/http` + `http.ServeMux`. В Go 1.22 в `ServeMux` добавлены method matching и wildcards, что снижает необходимость во внешних роутерах для ряда задач. citeturn2search3  
- **MUST выставлять server‑timeouts и лимиты заголовков** (защита от resource exhaustion и зависаний):  
  - `ReadHeaderTimeout` (предпочтительнее, чем “global ReadTimeout”, т.к. даёт handler’у контроль над телом запроса)  
  - `WriteTimeout`, `IdleTimeout`  
  - `MaxHeaderBytes` (по умолчанию `DefaultMaxHeaderBytes = 1MB`, но политики безопасности часто требуют меньшего) citeturn11view0turn11view1  
- **Graceful shutdown**: использовать `http.Server.Shutdown(ctx)` и дожидаться его завершения; `Shutdown` описан как корректный механизм graceful shutdown без прерывания активных соединений. citeturn4search1turn10view1  
- **Контейнерная среда и “disposability”**: ориентироваться на быстрый старт и корректный stop по SIGTERM/termination‑grace‑period; это согласуется и с 12‑factor “Disposability”, и с практикой termination grace period в orchestrator’ах. citeturn18search1turn4search4turn4search16  
- **TLS/HTTPS**: публичные REST endpoints должны быть доступны только по HTTPS; это базовая рекомендация OWASP. (Если TLS терминируется на ingress/sidecar — сервис всё равно должен понимать, что канал “внешний” защищён политикой платформы, и доверять заголовкам типа `X-Forwarded-*` только от trusted proxy.) citeturn4search3turn1search3  

### Наблюдаемость: логирование и OpenTelemetry

- **Структурные логи по умолчанию**: стандартный `log/slog` (Go 1.21+) как baseline. Это снижает dependency surface и облегчает LLM генерацию (меньше вариантов). citeturn0search2turn18search2  
- **OpenTelemetry по умолчанию** как vendor‑neutral стандарт для traces/metrics/logs; официальный “Go” раздел описывает генерацию и сбор сигналов. citeturn1search0turn1search16turn13search15  
- **Контекстная корреляция**: использовать propagation для сквозной корреляции сигналов между сервисами; OpenTelemetry прямо описывает, что context propagation позволяет связывать traces/metrics/logs across service boundaries. citeturn1search1turn7search1  
- **Collector как default маршрутинг/процессинг телеметрии**: OpenTelemetry Collector — vendor‑agnostic компонент для receive/process/export телеметрии, с pipeline‑моделью receivers/processors/exporters. citeturn20search6turn20search2turn20search22  
- **Semantic conventions**: в ручной instrumentation придерживаться семантических конвенций HTTP (атрибуты/спаны), чтобы данные были совместимы с тулзами и backend’ами. citeturn13search15turn13search7turn13search3  

### Конфигурация и секреты

- **Config через environment variables** как boring default (12‑factor): это снижает риск случайного коммита секретов, упрощает смену конфигурации между deploy’ами и соответствует OS‑agnostic стандарту. citeturn18search0turn9search17  
- **Логи в stdout/stderr** (а не в файлы) и “log stream” как responsibility окружения (12‑factor logs). citeturn9search0turn18search4  

### Безопасность API и dependency hygiene

- **Опираться на OWASP API Security Top 10 (2023)** как baseline threat taxonomy для API (в т.ч. object‑level authorization/BOLA). citeturn1search3turn1search7  
- **Инъекции**: проектировать доступ к БД так, чтобы исключить SQL injection (parameterized queries, no string concatenation). Это соответствуют рекомендациям OWASP cheat sheets. citeturn9search1turn9search4  
- **SSRF**: если сервис делает исходящие HTTP вызовы на URL из пользовательского ввода — применять защиты по OWASP SSRF Prevention (deny‑by‑default и строгая валидация/allowlist). citeturn9search2turn9search11  
- **Dependency vulnerability scanning**: `govulncheck` как официальный инструмент Go vulnerability management; он низкошумный и опирается на Go vulnerability database. citeturn0search3turn0search9turn0search12  
- **Go module integrity**: понимать, что `go` command по умолчанию может скачивать модули через `proxy.golang.org` и проверять через `sum.golang.org` checksum database; для enterprise‑среды это фиксируется политиками `GOPROXY/GOSUMDB`. citeturn3search11turn3search2turn3search5  

### Контейнерный образ и runtime hardening

- **Минимизировать attack surface**: distroless образы содержат только приложение и runtime‑зависимости, без shell/package manager. Это полезный boring default при нормальной observability/операторских практиках (с оговоркой: дебаг внутри контейнера усложняется). citeturn14search2turn14search6  
- **Запуск не‑root**: по умолчанию контейнер без `USER` запускается как root; это считается security‑risk. citeturn14search4  
- **Kubernetes security context**: использовать securityContext и Pod Security Standards как baseline ограничений (например, `runAsNonRoot`). citeturn14search1turn14search5  

### Testing и качество

- **Официальные рекомендации по стилю и тестам**: Effective Go + Code Review Comments + Test Comments. Эти документы дают “идиоматическую базу”, к которой можно привязать LLM‑правила и PR‑чеклисты. citeturn0search0turn0search1turn12search0  
- **Фаззинг** как дополнительный слой (особенно на парсинг/валидаторы): Go fuzzing официально описан как coverage‑guided и полезный для нахождения багов/уязвимостей. citeturn12search2turn12search5  
- **Тестовый фреймворк**: стандартный `testing` + `go test`. Официальная документация фиксирует conventions для `TestXxx(*testing.T)`. citeturn12search1turn12search17  

## Decision matrix и trade-offs

Ниже — “decisions you must make once” для template, чтобы LLM не гадал. В matrix — дефолт плюс альтернативы и их последствия.

### API транспорт: HTTP/JSON vs gRPC

| Критерий | HTTP/JSON (default) | gRPC |
|---|---|---|
| Совместимость | Максимальная, легко дергать curl/браузером | Отлично для service‑to‑service, строгие контракты |
| Контракты | OpenAPI/ручные DTO, риск drift | Protobuf как источник истины |
| Наблюдаемость | Отлично с OTel HTTP semconv | Отлично, gRPC имеет встроенные/стандартизируемые hooks |
| Ретраи | Обычно на клиенте; важно различать идемпотентность | Есть guidance по retries/backoff; важно понимать retryable операции citeturn13search1 |
| Когда выбрать | публичное API, BFF, простые сервисы | внутренние microservices, высокие требования к схемам/типам |

gRPC — проект CNCF (incubating) и позиционируется как high‑performance RPC framework. citeturn15search2turn15search14  

**Практический boring‑default** для универсального template: стартовать с HTTP (stdlib), но заложить расширяемость: отдельный пакет `internal/grpcapi` может быть добавлен без изменения domain/storage. Это снижает сложность initial scaffold, но не блокирует рост.

### Роутинг: stdlib ServeMux vs сторонний router

- **Default**: `net/http` `ServeMux` с паттернами (Go 1.22+). Это “standard way” и уже закрывает типовые маршруты через method+wildcards. citeturn2search3turn10view1  
- **Trade‑off**: если нужны middleware‑цепочки/подмаршруты/сложные matchers — сторонние роутеры (chi, etc) удобнее, но увеличивают dependency surface и расплывчатость для LLM (модель начинает “выбирать из множества”). Рекомендация: разрешать router как опцию только через ADR.

### Наблюдаемость: OpenTelemetry vs “Prometheus‑only” метрики

- **Default**: OpenTelemetry как единый стандарт сигналов (traces/metrics/logs) + Collector pipeline. citeturn1search16turn20search6turn20search22  
- **Trade‑off**: если инфраструктура — строго scrape‑based Prometheus и нет OTel pipeline, Prometheus‑клиент может быть проще. Но это снижает переносимость и ломает единую модель сигналов. (Решение фиксировать в ADR.)

### Конфигурация: env‑only vs env+файлы

- **Default**: env‑only в рантайме, в .env — только для локальной разработки. 12‑factor прямо рекомендует env vars как способ хранения конфигурации/секретов и stdout logs. citeturn18search0turn9search0  
- **Trade‑off**: если требуется сложная конфигурация (большие списки/структуры) — допустим config file, но его location должен задаваться env’ом и секреты не должны попадать в репозиторий; это отдельный ADR.

### Supply chain: “минимум” vs “полный DevSecOps”

- **Default минимум**: `govulncheck` + фиксируемость зависимостей через modules + checksum database, плюс строгий запрет на секреты в репо. citeturn0search9turn3search11turn18search0  
- **Более строгий режим**: SLSA provenance/attestations (через entity["organization","OpenSSF","open source security foundation"]‑проект SLSA) и sbom. Это повышает защищённость supply chain, но увеличивает pipeline complexity; подходит не всем командам. citeturn15search1turn14search3turn15search0turn15search6  

## LLM-правила и примеры MUST / SHOULD / NEVER

Ниже — “инструкция для LLM”, которую имеет смысл положить в `docs/llm/` и использовать как общий префикс/конституцию проекта. Формулировки сделаны так, чтобы уменьшить шанс “галлюцинаций” и сделать код детерминированным.

### MUST

1) **MUST следовать официальным Go рекомендациям по стилю, ошибкам и тестам**: Effective Go + Code Review Comments + Test Comments. citeturn0search0turn0search1turn12search0  

2) **MUST прокидывать `context.Context` через весь request/operation pipeline** (handler → domain → storage → outbound calls) и уважать cancellation/deadlines. Контекст предназначен для дедлайнов/отмены и request‑scoped values; цепочка вызовов должна его передавать. citeturn7search1turn7search18turn1search1  

3) **MUST использовать структурное логирование через `log/slog`**, не “ручные printf‑логи”. Structured logs нужны для поиска/фильтрации в production. citeturn0search2turn18search2  

4) **MUST настраивать `http.Server` безопасными параметрами** в шаблоне сервиса:
- `ReadHeaderTimeout`, `WriteTimeout`, `IdleTimeout`, `MaxHeaderBytes` (минимальный baseline)  
- корректный graceful shutdown через `Server.Shutdown(ctx)`  
Всё это прямо описано в документации `net/http`. citeturn11view1turn10view1turn4search1  

5) **MUST разделять пакеты по слоям так, чтобы транспорт не проникал в domain**, а внешние зависимости были сконцентрированы в “краях” (`cmd/`, `internal/app/`, `internal/storage/`, `internal/httpapi/`). Это продолжает идею официального module layout: `internal` как зона быстрой эволюции. citeturn17view0turn8search4  

6) **MUST писать код так, чтобы сервис был disposable**: быстрый старт, корректный stop, без потери in‑flight операций в пределах grace period. citeturn18search1turn4search4turn10view1  

7) **MUST использовать `govulncheck` и реагировать на найденные уязвимости**, включая транзитивные зависимости. citeturn0search9turn0search3  

### SHOULD

1) **SHOULD использовать OpenTelemetry для трасс/метрик** и придерживаться semantic conventions, чтобы телеметрия была совместима с backend’ами и collector pipeline. citeturn1search0turn13search15turn20search22  

2) **SHOULD реализовывать retries только там, где операция безопасна для повторения**, и применять экспоненциальный backoff с jitter для фоновых операций (избегая retry‑storm). Транзиентные ошибки и backoff‑with‑jitter — стандартная рекомендация. citeturn13search0turn13search4turn13search1  

3) **SHOULD использовать env‑config как primary source**, а `.env` — только как локальную помощь; это соответствует 12‑factor. citeturn18search0turn9search17  

4) **SHOULD избегать вводить новые внешние зависимости без ADR**: каждое добавление dependency — это supply‑chain risk и maintenance cost (a.k.a. “boring” принцип). Базово Go modules задизайнены под воспроизводимые сборки и контроль зависимостей. citeturn3search1turn3search11turn8search19  

5) **SHOULD использовать `internal/` как default место для пакетов**, делая публичными только действительно нужные экспортируемые части (если это вообще не библиотека). citeturn17view0turn8search4  

### NEVER

1) **NEVER использовать `context.Background()` / `context.TODO()` в середине business‑операции**, если контекст уже существует. Контекст должен прокидываться по цепочке (см. docs). citeturn7search1turn7search18  

2) **NEVER отключать или игнорировать server timeouts/лимиты** в HTTP сервере. Нулевые значения часто означают “no timeout”. Документация `net/http` прямо описывает семантику “zero means no timeout” для ряда полей. citeturn11view1turn10view0  

3) **NEVER логировать секреты/токены/пароли и не хранить их в репозитории**. 12‑factor отдельно отмечает, что env‑config снижает шанс случайного коммита секретов. citeturn18search0  

4) **NEVER строить SQL запросы конкатенацией строк из пользовательского ввода**. Это прямой класс инъекций, против которого OWASP рекомендует параметризацию и безопасные практики. citeturn9search1turn9search4  

5) **NEVER доверять произвольным URL для исходящих запросов без SSRF‑защит**. SSRF — отдельный класс угроз; придерживаться OWASP SSRF Prevention. citeturn9search2turn9search11  

### Concrete good / bad examples на Go

**Good: безопасная настройка HTTP сервера + graceful shutdown**

```go
package main

import (
	"context"
	"log/slog"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

	mux := http.NewServeMux()
	mux.HandleFunc("GET /livez", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})
	mux.HandleFunc("GET /readyz", func(w http.ResponseWriter, r *http.Request) {
		// TODO: check dependencies that are REQUIRED to serve traffic.
		w.WriteHeader(http.StatusOK)
	})

	srv := &http.Server{
		Addr:              ":8080",
		Handler:           mux,
		ReadHeaderTimeout: 5 * time.Second,
		WriteTimeout:      30 * time.Second,
		IdleTimeout:       60 * time.Second,
		MaxHeaderBytes:    1 << 20, // consider lowering based on policy
	}

	ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
	defer stop()

	errCh := make(chan error, 1)
	go func() {
		logger.Info("http server starting", "addr", srv.Addr)
		errCh <- srv.ListenAndServe()
	}()

	select {
	case <-ctx.Done():
		logger.Info("shutdown requested")
	case err := <-errCh:
		logger.Error("server stopped unexpectedly", "err", err)
	}

	shutdownCtx, cancel := context.WithTimeout(context.Background(), 20*time.Second)
	defer cancel()

	if err := srv.Shutdown(shutdownCtx); err != nil {
		logger.Error("graceful shutdown failed", "err", err)
	}
	logger.Info("shutdown complete")
}
```

Почему это “good”: использует pattern‑routing `ServeMux` (Go 1.22+), включает ключевые таймауты/лимиты, применяет `Server.Shutdown`, и готов к Kubernetes termination semantics. citeturn2search3turn11view1turn10view1turn4search4turn4search16  

**Bad: “нулевые таймауты”, отсутствие shutdown и контекста**

```go
// Плохой пример: зависания соединений, нет лимитов, нет shutdown.
http.ListenAndServe(":8080", handler)
```

Это проблемно, потому что без явных server settings легко получить “no timeout” поведение и неконтролируемые подвисания; документация `net/http` прямо описывает, что нулевые значения часто означают “нет таймаута”. citeturn11view1turn10view0turn10view1  

**Good: прокидывание context в исходящий HTTP вызов**

```go
req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
if err != nil {
	return err
}
resp, err := http.DefaultClient.Do(req)
```

`net/http` прямо рекомендует `NewRequestWithContext` + `Client.Do` для context‑aware HTTP. citeturn7search7turn7search1  

## Anti-patterns и типичные ошибки моделей

Здесь “anti‑patterns” сформулированы как *запрещённые или опасные конструкции*, которые LLM нередко склонна генерировать “по привычке” из других языков/фреймворков. Каждый пункт привязан к источнику, который объясняет, почему это неправильная default‑практика.

### Транспорт и жизненный цикл

- **Anti‑pattern: readiness probe проверяет внешние зависимости (DB/broker) и используется как liveness** → это ведёт к cascade restarts. Kubernetes различает liveness (когда надо рестартить контейнер) и readiness (когда не надо слать трафик). Liveness failures вызывают рестарт. citeturn1search6turn1search2  
- **Anti‑pattern: отсутствие graceful shutdown** → прерывание in‑flight запросов и некорректное завершение в orchestrated среде. `Server.Shutdown` предназначен для graceful shutdown. citeturn4search1turn10view1turn18search1  

### Context и наблюдаемость

- **Anti‑pattern: бизнес‑данные в `context.WithValue`**. Контекст предназначен для request‑scoped values, deadlines и cancellation; злоупотребление `WithValue` ухудшает читаемость и ломает контракт “context as control plane”. citeturn7search1turn7search18  
- **Anti‑pattern: несогласованная телеметрия (нет propagation, разные атрибуты)** → traces/metrics/logs плохо коррелируются. OTel подчёркивает роль context propagation для кросс‑сервисной корреляции. citeturn1search1turn13search15  

### Security

- **Anti‑pattern: “HTTP внутри кластера ок” без чёткой модели доверия**. OWASP REST Security требует HTTPS endpoints; если TLS завершается на ingress, это должно быть явно зафиксировано как trust boundary (ADR), чтобы не превращать “внутренний HTTP” в случайную дыру. citeturn4search3  
- **Anti‑pattern: SQL конкатенацией** → SQL injection. citeturn9search1turn9search4  
- **Anti‑pattern: “fetch remote URL” без SSRF‑защит** (allowlist, блок внутренних адресов, deny‑by‑default). citeturn9search2turn9search11  
- **Anti‑pattern: запуск контейнера как root по умолчанию**. В Docker без `USER` по умолчанию root; это риск. citeturn14search4turn14search5  

### Надёжность и retries

- **Anti‑pattern: ретраи “на всё подряд”** → retry storm, ухудшение деградации. Guidance по transient faults рекомендует backoff+jitter и внимательный выбор операций для retry. citeturn13search4turn13search0turn13search1

## Консистентность данных и распределённые транзакции

Цель секции — дать формализуемые правила, чтобы LLM могла проектировать multi‑step business flow между сервисами без 2PC/dual writes, с контролируемыми race conditions.

### Базовый принцип: transactional boundaries = внутри сервиса

В микросервисной архитектуре каждый сервис обычно владеет собственной БД; пересечение границы сервиса = пересечение границы транзакции. В результате традиционные ACID‑гарантии не распространяются на весь бизнес‑workflow, и приходится управлять консистентностью на уровне протокола/процесса. citeturn5search1turn19view2turn19view3  

Классический аргумент против “глобальных” распределённых транзакций: протоколы типа 2PC могут блокировать прогресс при недоступности узлов и часто оказываются хрупкими/дорогими в больших системах. Это подчёркивается в позиционном докладе entity["people","Pat Helland","distributed systems researcher"] о том, почему индустрия часто “отказывается” от distributed transactions в пользу других паттернов. citeturn19view0turn19view2turn19view3  

Даже когда СУБД поддерживает 2PC, документация подчёркивает, что это предназначено для внешних transaction managers (XA), а не для “обычного приложения”. Например, PostgreSQL описывает 2PC как механизм, предназначенный для внешних систем управления транзакциями. citeturn6search17turn6search1  

**Практическое следствие для template и LLM**: *каждый шаг workflow должен быть local transaction в одном сервисе; межсервисная координация — через сообщения/события и компенсации.*

### Saga: orchestration vs choreography и когда что выбирают

Saga — дизайн‑паттерн, который обеспечивает консистентность в распределённой системе, разбивая процесс на последовательность **локальных транзакций** с **компенсирующими транзакциями** на случай ошибок. citeturn5search1turn5search14  

**Orchestration saga**: есть центральный orchestrator, который знает шаги и явно говорит участникам, что делать. AWS описывает orchestrator как coordinator всего lifecycle и обмена сообщениями “вперёд/назад”. citeturn5search0turn5search7  

**Choreography saga**: сервисы реагируют на события друг друга (event subscriptions), без центрального координатора. AWS описывает choreography как механизм сохранения целостности через подписки на события. citeturn5search4  

**Trade‑offs** (важно формализовать в ADR, иначе LLM будет “угадывать”):

- Orchestration: проще наблюдать end‑to‑end workflow, легче централизовать retries/timeouts/policies, но появляется orchestrator как “центр сложности” и потенциальное место для ошибок/узкого места. citeturn5search0turn5search7  
- Choreography: меньше централизованного контроля, потенциально сильнее слабая связность, но сложнее понимать поток, дебажить и предотвращать “event‑циклы”. citeturn5search4  

**Boring default для greenfield**: orchestration для сложных multi‑step бизнес‑процессов (где важен контроль, аудит, наблюдаемость), choreography — только для простых/локальных реакций или когда домен действительно “event‑first”.

### Компенсации, pivot-транзакция и идемпотентность

Ключевая инженерная сложность sаga — отсутствие “автоматического rollback”. Microsoft подчёркивает, что при ошибке saga выполняет компенсирующие транзакции, чтобы “откатить” завершённые шаги. citeturn5search1turn5search14  

Microsoft также вводит практическую идею **pivot transaction**: граница между undoable/compensable шагами и шагами, которые должны быть retryable (идемпотентные), чтобы saga “доползла” до финального состояния. citeturn5search1  

Из доклада Helland следует ещё один практический факт: в больших системах сообщения часто доставляются “at‑least‑once”, возможны retries и out‑of‑order arrival; значит, получатели должны быть *идемпотентны* и помнить обработанные сообщения. citeturn19view2turn19view3  

**Идемпотентность на уровне API**: Stripe описывает, как idempotency key позволяет серверу распознать повтор запроса и вернуть тот же результат (включая ошибки). Это хороший “authoritative” эталон поведения для POST‑операций. citeturn6search0turn6search12  

### Dual writes и Outbox pattern

Одна из самых опасных ловушек — **dual writes**: “в одной транзакции записали в БД, а потом отправили событие/сообщение” (или наоборот). Между этими действиями возможен сбой, приводящий к расхождениям.

**Outbox pattern** решает проблему: сервис в рамках одной local transaction записывает и бизнес‑изменение, и запись в outbox‑таблицу; далее CDC/публикатор надёжно превращает outbox‑строки в события для других сервисов. Debezium прямо формулирует outbox как способ “безопасно и надёжно обмениваться данными” и избегать несогласованности между внутренним состоянием сервиса и событиями, потребляемыми другими сервисами. citeturn5search2turn5search15  

**Boring default**: если сервис публикует доменные события наружу — *outbox обязателен* (или эквивалентная гарантированная атомарность state+event), иначе консистентность будет случайной.

### Read models, CQRS и reconciliation jobs

CQRS разделяет модели чтения и записи: commands меняют write‑модель, а queries читают read‑модель, которая может быть оптимизирована отдельно. Microsoft определяет CQRS как разделение read/write операций и моделей. citeturn6search2  

Event sourcing / projections часто означают **eventual consistency**: материализованные представления обновляются с задержкой, что прямо отмечается в документации (создание projections приводит к eventual consistency). citeturn6search6  

**Reconciliation jobs** (сверочные фоновые процессы) становятся практической необходимостью в eventual consistency системах:  
- если события могли быть пропущены, обработаны дважды или out‑of‑order, reconciliation восстанавливает read‑модель и “закрывает хвосты” (сопоставимо с идеей Helland о необходимости терпеть retries/out‑of‑order и проектировать обработку сообщений). citeturn19view2turn19view3turn6search6  

**Boring default**: каждое событийное/асинхронное взаимодействие должно иметь:
- дедупликацию/идемпотентность у consumer’а  
- наблюдаемое DLQ/parking lot для “poison” сообщений  
- reconciliation job или периодическую проверку инвариантов (для критичных доменных данных)

### Distributed locks: когда нужны и почему опасны

В микросервисах распределённые locks — частая “ложная кнопка”. Если lock нужен “для корректности”, нужно понимать failure modes (pauses, clock, partitions) и использовать fencing tokens.

- Redis документирует distributed locks и алгоритм Redlock, но вокруг корректности Redlock существует известная дискуссия. citeturn6search23turn6search11  
- Ключевой практический тезис из анализа Kleppmann: если lock нужен для correctness, нельзя полагаться на наивные схемы; требуется consensus‑система и **fencing tokens**. citeturn6search3turn16search5  
- Даже у entity["organization","etcd","distributed key-value store"] в issue подчёркнуто: “нельзя использовать etcd как наивную locking system”; нужно тщательно связывать fencing token (например, revision) с системой, которую защищаем. citeturn16search0  

**Boring default**: избегать distributed locks как средства бизнес‑согласованности между сервисами. Предпочитать:
- optimistic concurrency (версии/ETag)  
- идемпотентность  
- саги/очереди  
Lock — только для технических задач и только с fencing token + чёткими границами.

### Формализованные правила для LLM: как проектировать multi-step business flow

Эти правила предназначены для включения в `docs/llm/instructions.md` и `docs/decisions/adr-0003-data-consistency.md`.

#### MUST

- **MUST ставить границу транзакции на уровне одного сервиса и одной БД**: каждый шаг — local transaction. Идея “atomic transactions cannot span entities” и разделение scopes подчёркнуты в работе Helland. citeturn19view2turn19view3  
- **MUST проектировать межсервисные workflows как saga** (orchestrated по умолчанию), с явными compensating actions для каждого “undoable” шага. citeturn5search1turn5search0turn5search14  
- **MUST делать все команды/обработчики событий идемпотентными** (из-за at‑least‑once доставки и retries). Helland описывает retries/out‑of‑order и необходимость idempotence на стороне получателя. citeturn19view2turn19view3  
- **MUST использовать idempotency keys для внешних команд (HTTP/gRPC) там, где клиент может retry’ить**; поведение key‑based дедупликации хорошо формализовано Stripe. citeturn6search0turn6search12  
- **MUST избегать dual writes**: если сервис публикует события/сообщения о commit’е — использовать outbox (или эквивалент), чтобы состояние и событие не расходились. citeturn5search2turn5search15  

#### SHOULD

- **SHOULD выбирать orchestration для бизнес‑критичных flows** (нужны наблюдаемость, контроль, повторяемость шагов). AWS/ Microsoft описывают оркестрацию как централизованный контроль жизненного цикла. citeturn5search0turn5search1turn5search7  
- **SHOULD использовать CQRS/read models**, когда нагрузка на чтение и требования к query‑формам отличаются от write‑модели; осознавать delay/eventual consistency. citeturn6search2turn6search6  
- **SHOULD стандартизировать event envelope** (например, CloudEvents) для интероперабельности между брокерами/сервисами. CloudEvents — спецификация CNCF для общего формата event metadata. citeturn16search12turn16search2turn16search3  

#### NEVER

- **NEVER внедрять 2PC как default решение для межсервисной консистентности**. 2PC предназначен для внешних transaction managers и может приводить к блокировкам/хрупкости; индустриальная практика часто уходит в сторону альтернатив (saga, messaging). citeturn6search17turn6search1turn19view2turn5search1  
- **NEVER использовать распределённые locks для корректности без fencing tokens и чёткого анализа failure modes**; для etcd это прямо проговорено как “locks aren’t really locks” без fencing. citeturn16search0turn6search3turn6search23

**Outbox (псевдо‑код на Go): local transaction = бизнес‑изменение + outbox row**

```go
type OutboxEvent struct {
	ID        string
	Type      string
	BodyJSON  []byte
	CreatedAt time.Time
}

func (r *Repo) CreateOrder(ctx context.Context, order Order) error {
	return r.withTx(ctx, func(tx *sql.Tx) error {
		if err := r.insertOrder(ctx, tx, order); err != nil {
			return err
		}
		ev := OutboxEvent{
			ID:        uuid.NewString(),
			Type:      "order.created",
			BodyJSON:  mustJSON(orderCreatedPayload(order)),
			CreatedAt: time.Now().UTC(),
		}
		if err := r.insertOutbox(ctx, tx, ev); err != nil {
			return err
		}
		return nil
	})
}
```

Смысл: outbox pattern предотвращает несогласованность между локальным состоянием и публикуемыми событиями. citeturn5search2turn5search15  

**Saga orchestrator (упрощённый скелет):**

```go
type Step interface {
	Do(ctx context.Context, sagaID string) error
	Compensate(ctx context.Context, sagaID string) error
}

func RunSaga(ctx context.Context, sagaID string, steps []Step) error {
	var done []Step
	for _, s := range steps {
		if err := s.Do(ctx, sagaID); err != nil {
			// компенсации в обратном порядке
			for i := len(done) - 1; i >= 0; i-- {
				_ = done[i].Compensate(ctx, sagaID)
			}
			return err
		}
		done = append(done, s)
	}
	return nil
}
```

Это иллюстрирует базовый принцип: saga = последовательность local transactions + compensation при ошибке. citeturn5search1turn5search0turn5search14
