# Engineering standard и LLM-instructions для production-ready Go microservice template

## Scope

Этот стандарт применим, когда вы делаете **новый (greenfield) микросервис** на Go в стиле “boring, battle‑tested defaults”: небольшой/средний сервис с HTTP/gRPC API, несколькими внешними зависимостями (БД/кеш/внешние HTTP‑сервисы), деплоем в контейнеры (часто Kubernetes), и требованиями к наблюдаемости, безопасности, управляемости и предсказуемой деградации. Он специально ориентирован на сценарий “клонировал репозиторий → сразу пишу код” и на то, чтобы LLM-инструменты генерировали **идиоматичный, безопасный, поддерживаемый Go‑код**, опираясь не на догадки, а на закреплённые решения и явные контракты. citeturn22view0turn9view0turn15view0turn6search2

Подход **не** стоит применять “как шаблон для всего”, если:
- сервис — это **высоконагруженный latency‑critical** компонент, где нужна тонкая ручная оптимизация аллокаций/GC/системных вызовов; в таком случае стандарт должен быть дополнен специфичными performance‑ADR и профилированием, иначе “универсальные дефолты” могут мешать. citeturn17search2turn5search0  
- у вас уже есть зрелая платформа (service mesh, стандартные sidecar‑политики retries/CB/rate limiting, единый API gateway), и внедрение тех же механизмов в коде **будет дублировать** инфраструктурный слой и усложнять отладку. При этом базовые требования (таймауты, контексты, корректное завершение, наблюдаемость) остаются обязательными даже в presence of mesh. citeturn4search2turn3search4turn24view0turn8view1  
- сервис — **скрипт/CLI/одноразовая утилита**, где требования production‑эксплуатации (пробы, graceful shutdown, OTEL, retry budgets) не окупаются. citeturn22view0turn10search1  

Ключевой принцип: стандарт задаёт “рельсы” и безопасные умолчания, но **не заменяет** SLO‑ориентированную настройку (таймауты, лимиты, ретраи, деградация) и архитектурные решения, фиксируемые в ADR. citeturn4search17turn4search0turn4search12  

## Recommended defaults для greenfield template

### Базовая платформа и версия Go

- **Go version policy**: шаблон должен целиться в **текущую поддерживаемую** ветку Go (на дату 2026‑03‑02 это Go 1.26.x как последний major релиз; релиз 1.26.0 датирован 2026‑02‑10) и допускать сборку на предыдущей поддерживаемой (1.25.x), потому что политика Go: major релиз поддерживается, пока не выйдет два более новых major релиза. citeturn9view0turn0search12  
- **В go.mod** фиксировать `go 1.26` (или минимум вашей поддерживаемой линии) и документировать поддержку явно в README/COMPATIBILITY. Это уменьшает “догадки” LLM и разработчиков о допустимых фичах стандартной библиотеки. citeturn9view0turn13search3  

### Layout репозитория (минимум магии, максимум предсказуемости)

Рекомендуемый “boring” layout (сознательно небольшой):

- `cmd/<service>/main.go` — только wiring: конфиг, логгер, телеметрия, сбор зависимостей, старт/стоп.  
- `internal/` — вся бизнес‑логика и адаптеры; использовать механизм `internal/` для контроля публичной поверхности и исключения “случайной библиотеки наружу”. Механизм `internal` закреплён в Go (в т.ч. enforcement описан в Go 1.4+ и деталях дизайна). citeturn2search0turn2search7turn2search3  
- `pkg/` — **только** если вы реально предоставляете переиспользуемую библиотеку другим репозиториям; иначе избегать, чтобы LLM не “вытаскивала наружу” внутренние детали. (Это правило — инженерная конвенция, но напрямую подкрепляется назначением `internal/` как механизма снижения public API surface.) citeturn2search7turn2search0  
- `docs/` — стандарты, ADR, runbooks (см. отдельную секцию “что вынести в файлы”).  
- `configs/` или `deploy/` — примеры манифестов и конфигов (Kubernetes/Compose), но без “production‑секретов”. Практики работы с Secrets явно описывать и ссылаться на Kubernetes guidance. citeturn6search3turn6search7  

### Стиль и базовые правила Go-кода (как норматив)

- Код **всегда** форматируется `gofmt`; это стандартный инструмент и явно рекомендован как минимум для устранения “механических” стилевых проблем (и де‑факто почти весь Go‑код в экосистеме ему следует). citeturn21search0turn22view0  
- Контексты:
  - `context.Context` **передаётся по цепочке вызовов** от входящего запроса к исходящим, и функции обычно принимают `ctx` **первым параметром**; хранить `Context` в struct — запрещено (кроме случаев подгонки под внешний интерфейс). citeturn22view0turn1search1  
- Ошибки:
  - Не использовать `panic` для обычного control flow; ошибки возвращать явными значениями. citeturn22view0turn6search1  
  - Для обёртки ошибок использовать `fmt.Errorf(... %w ...)`, чтобы работали `errors.Is/As` (стандартная модель wrapping). citeturn21search3  

### HTTP сервер по умолчанию (если сервис — HTTP API)

Шаблон должен запускать `http.Server` **с явными таймаутами**, потому что нулевые значения означают “без таймаута”, и Go прямо поясняет, что `ReadHeaderTimeout` обычно предпочтительнее `ReadTimeout`, так как позволяет обработчику принимать решения по телу запроса. citeturn7view0turn24view0  

Минимально обязательные поля `http.Server` в шаблоне:
- `ReadHeaderTimeout` (защита от медленных заголовков/Slowloris‑класса проблем),  
- `IdleTimeout` (ограничение keep‑alive простоя),  
- опционально `WriteTimeout` (осторожно для streaming/long‑poll).  
Также задать `MaxHeaderBytes`/лимиты тела (см. ниже). citeturn7view0turn14search0  

### HTTP client (outgoing) по умолчанию: reuse + таймауты + лимиты соединений

- `http.Client` должен быть **переиспользуемым**, потому что `Transport` кэширует соединения; документация net/http прямо говорит, что Clients/Transports нужно reuse, они thread‑safe, и новый клиент “на каждый вызов” приводит к лишним подключениям/ресурсам. citeturn8view1turn8view2  
- Глобальный `Client.Timeout` задаёт верхнюю границу на весь запрос (включая коннект/редиректы/чтение body) и отменяет запрос как будто завершился контекст запроса. citeturn8view1turn23view0  
- Для bulkhead‑эффекта на уровне клиента использовать лимиты `Transport.MaxConnsPerHost` и/или семафоры на уровень зависимости; net/http документирует `MaxConnsPerHost` как лимит total connections per host, при превышении dials будут блокироваться. citeturn24view0turn11search1  

### Валидация входа и защита от resource consumption по умолчанию

- Лимитировать размер request body:
  - использовать `http.MaxBytesReader`, потому что он предназначен именно для предотвращения случайной/злонамеренной отправки слишком больших данных и траты ресурсов сервера. citeturn14search0  
- “Строгий JSON” для API по умолчанию:
  - использовать `json.Decoder.DisallowUnknownFields()` при декодировании в struct, чтобы неизвестные поля были ошибкой (снижает риск “тихих” ошибок клиента и неправильных контрактов, а также уменьшает свободу LLM “добавить поля как получится”). citeturn14search1  
- В ответах не раскрывать внутренние детали ошибок; требования к error handling/logging хорошо формализованы в ASVS (V7) и OWASP guidance по security logging. citeturn12search4turn12search1  
- Rate limiting / throttling — базовая защита от “Unrestricted Resource Consumption” в контексте API; OWASP API Security Top 10 2023 прямо рекомендует лимитировать частоту взаимодействия клиента в заданном окне времени и настраивать политики по бизнес‑нуждам. citeturn14search2turn3search7  

### Observability: OpenTelemetry как общий язык

- Для трассировки/метрик использовать OpenTelemetry SDK и придерживаться семантических конвенций (Semantic Conventions и Resource attributes), чтобы телеметрия была сопоставимой между сервисами. citeturn15view0turn16search6turn16search2  
- Для HTTP инструментировать handler’ы через официальную `otelhttp`‑обёртку. citeturn16search0  
- Для gRPC инструментировать через `otelgrpc` (StatsHandler). citeturn16search1  
- Логи:
  - базовый boring‑default: структурированные логи через `log/slog` (в стандартной библиотеке с Go 1.21). citeturn0search1  
  - экспорт логов через OpenTelemetry рассматривать как опцию, потому что в Go Getting Started по OpenTelemetry отмечает, что logs signal всё ещё experimental и возможны breaking changes. citeturn16search8turn15view0  
- Если вы используете OpenTelemetry Collector: уменьшать attack surface и конфигурационную сложность — официальная рекомендация “использовать только нужные компоненты”. citeturn16search3turn16search7  

### Конфигурация и секреты

- Конфигурацию ожидаем **из окружения** (env vars), а не из файлов в образе; в Kubernetes это естественно сочетается с ConfigMaps/Secrets. citeturn13search5turn6search7  
- Для логов — принцип “treat logs as event streams”: писать в stdout/stderr, а агрегацию отдавать платформе. citeturn13search0turn0search1  
- Для Secrets следовать Kubernetes “good practices”: это отдельный документ, который стоит прямо линковать из template. citeturn6search3  

### CI/качество/безопасность supply chain как минимум

Обязательный минимум PR‑гейтов в шаблоне:
- `gofmt` (и/или goimports) как механический стандарт. citeturn22view0turn21search0  
- `go test ./...` + отдельная цель `go test -race` в CI: race detector находит гонки, которые происходят во время выполнения, поэтому его полезность зависит от покрытия тестами, но он является стандартным инструментом и прямо документирован в Go. citeturn18search15turn18search0  
- `go vet ./...`: статический анализ suspicious constructs — часть официального toolchain. citeturn17search0turn17search18  
- `govulncheck ./...`: официальный low‑noise анализ уязвимостей, который “поднимает” только реально достижимые уязвимости по call graph и связан с Go vulnerability database. citeturn0search3turn0search7turn0search19  
- Опционально (в зависимости от регуляторики и критичности): зафиксировать цель на SLSA‑уровень (как минимум provenance/attestations), опираясь на OpenSSF/SLSA guidance. Важно: версии/статус спецификации меняются, поэтому ссылка и ADR обязательны. citeturn10search2turn10search5  

## Decision matrix и trade-offs

Ниже — матрица решений для “универсального, но boring” Go‑микросервиса. Здесь “Default” — то, что стоит включить в template; “Alternative” — допустимые варианты с условиями.

| Область | Default (template) | Alternative | Trade‑offs / когда менять |
|---|---|---|---|
| API транспорт | `net/http` + явные `http.Server` timeouts | gRPC | HTTP проще для публичных API и быстрее стартовать; gRPC дисциплинирует контракты и deadlines (gRPC поддерживает client‑side deadlines и сервер автоматически отменяет RPC по истечении дедлайна). citeturn7view0turn5search0turn16search1 |
| Observability | OpenTelemetry traces+metrics, `slog` для логов | OTEL logs pipeline | OTEL даёт общий формат/семантику; logs в OTEL для Go помечены как experimental → риск breaking changes, поэтому stdout+структурированные логи остаются boring default. citeturn15view0turn16search8turn0search1 |
| Retry | Малое число retries + backoff+jitter + retry budget | “наивные” ретраи, бесконечные ретраи | Ретраи могут взорвать нагрузку и вызвать каскадные сбои; SRE и Envoy подчёркивают идею retry budgets и ограничение взрывного объёма retries. citeturn4search1turn4search2turn4search0turn4search6 |
| Circuit breaker | По умолчанию — минимальный “breaker на retries” или лимит активных retries; полнофункциональный CB — по ADR | Полный CB в каждом клиенте | Envoy рекомендует retry budgets, а если делаете static circuit breaking — агрессивно ограничивать retries. Полный CB усложняет тюнинг и может ухудшить ситуацию при неправильных порогах. citeturn4search2turn11search9 |
| Bulkheads | Лимиты конкурентности per‑dependency (семафоры / `MaxConnsPerHost`) | Без bulkheads | Bulkhead pattern ограничивает blast radius; Microsoft прямо рекомендует использовать процессы/пулы/семафоры и сочетать с retry/CB/throttling. citeturn11search1turn24view0 |
| Rate limiting | Token bucket (`x/time/rate`) на входе (и/или per‑dependency) | Только на gateway | OWASP API Top 10 2023 рекомендует rate limiting как защиту от resource consumption; token bucket — стандартный подход, `x/time/rate` документирует limiter как token bucket. citeturn14search2turn11search0turn11search4 |

В Kubernetes preStop и terminationGracePeriodSeconds влияют на порядок/время остановки — это нужно учитывать. citeturn24view0turn10search1turn3search1turn3search5 |
| Kubernetes probes | `/live` (без внешних deps), `/ready` (минимальный набор критичных проверок), `startupProbe` при долгом старте | Один `/health` на всё | Kubernetes различает liveness/readiness/startup; liveness для рестарта (например deadlock), readiness для исключения из трафика; смешивание с внешними deps часто приводит к flapping и cascading проблемам. citeturn3search4turn3search0turn3search1 |
| Эволюция релиза | Canary/Blue‑Green через контроллер прогрессивного деплоя + feature flags | Big‑bang релизы | Canary снижает риск через постепенный rollout; Blue‑Green упрощает cutover/rollback; feature flags лучше стандартизировать через OpenFeature API. citeturn20search1turn20search4turn11search3turn11search2 |

## Правила для LLM: MUST / SHOULD / NEVER

Ниже — **нормативный набор правил**, который стоит оформить как “общий префикс” (system prompt) для LLM‑инструментов в репозитории. Цель: минимизировать галлюцинации, произвольный выбор библиотек и нарушение эксплуатационных требований.

### MUST

LLM **MUST**:

1) **Следовать существующим паттернам репозитория**: копировать стиль, структуру пакетов, способы конфигурирования, логирования, обработку ошибок и контекстов из уже присутствующих файлов, а не из “внешней памяти”. (Это принцип “reduce guesswork”: в Go стиле контексты/ошибки/форматирование и др. фиксируются практиками review и стандартными инструментами.) citeturn22view0turn21search0turn21search3  

2) **Пропускать `context.Context` через всю цепочку**: от inbound request до outbound calls, ставить дедлайны на внешние вызовы, корректно вызывать `cancel()` для derived contexts; контекст обычно должен быть первым параметром и не храниться в struct. citeturn22view0turn1search1turn8view1  

3) **Явно задавать таймауты**:
   - на HTTP сервере: использовать `ReadHeaderTimeout` и/или `ReadTimeout`/`WriteTimeout`/`IdleTimeout` осознанно; нулевые значения означают “без таймаута”. citeturn7view0turn24view0  
   - на HTTP клиенте: `Client.Timeout` и/или `Transport` таймауты (TLSHandshakeTimeout, ResponseHeaderTimeout и т.д.), и использовать per‑request context для жизненного цикла запроса. citeturn8view1turn24view0turn23view0  

4) **Переиспользовать http.Client/Transport** и закрывать `resp.Body` у исходящих запросов, иначе деградирует reuse соединений и растёт ресурсная нагрузка. citeturn8view1turn19view2turn8view2  

5) **Ограничивать потребление ресурсов**:
   - лимитировать request body через `http.MaxBytesReader` (или `MaxBytesHandler`). citeturn14search0  
   - использовать rate limiting на входе (token bucket) как базовую защиту от Unrestricted Resource Consumption. citeturn14search2turn11search0  

6) **Логировать структурировано** через `log/slog` и избегать утечек чувствительных данных; опираться на OWASP logging guidance. citeturn0search1turn12search1turn6search3  

7) **Соблюдать контроль качества**: код должен проходить `gofmt`, `go test`, `go vet` и `govulncheck` (если включены в репозитории). citeturn22view0turn17search0turn0search7  

### SHOULD

LLM **SHOULD**:

1) Делать изменения **маленькими и ревью‑дружелюбными**, добавляя тесты там, где это разумно; `go test` — стандартный способ запуска тестов. citeturn18search5turn18search1  

2) Для ошибок использовать wrapping и сохранение причины (`%w`), чтобы вызывающий код мог различать классы ошибок через `errors.Is/As`. citeturn21search3turn6search1  

3) Делать обработку JSON строгой (`DisallowUnknownFields`), если сервис позиционируется как контрактный API и требуется early‑fail на несовпадение схемы. citeturn14search1  

4) При реализации retries:
   - ретраить **только** безопасные/идемпотентные операции; HTTP семантика предупреждает, что клиент не должен автоматически ретраить non‑idempotent методы без уверенности, что семантика идемпотентна. citeturn5search3turn5search7  
   - использовать backoff+jitter и лимиты (budget), потому что “взрыв retries” — известная причина каскадных отказов; это отражено в SRE guidance и в рекомендациях Envoy про retry budgets. citeturn4search1turn4search2turn4search6turn4search0  

5) Если нужен feature flagging — завести абстракцию, совместимую с OpenFeature, чтобы не привязываться к конкретному вендору. citeturn11search2turn11search6  

### NEVER

LLM **NEVER**:

1) **Не добавлять новые зависимости** (модули, фреймворки, middleware) без явной причины, ADR и обновления docs. В противном случае модель начинает “тащить привычное”, конфликтуя с boring default и снижая воспроизводимость. (Практический контроль: go.mod/go.sum меняются только вместе с ADR и rationale.) citeturn13search3turn22view0  

2) **Не использовать `panic` для обычных ошибок** и не превращать ошибки в “просто log + continue”, если это нарушает контракт/целостность. citeturn22view0turn6search1  

3) **Не создавать `context.Context` “глобально” и не хранить в struct**, не использовать кастомные Context‑типы. citeturn22view0turn1search1  

4) **Не делать бесконечные/агрессивные ретраи** и не ретраить non‑idempotent вызовы “на авось” — это типовой триггер каскадных падений. citeturn5search7turn4search0turn4search2  

5) **Не логировать секреты/токены/пароли**, и не включать “debug‑логирование всего запроса/ответа” в прод‑код без redaction. OWASP подчёркивает важность корректного security logging и контроля содержимого логов, а Kubernetes отдельно формулирует практики работы с Secrets. citeturn12search1turn6search3  

### Добавка к общему префиксу: resilience patterns (обязательные установки)

Эту часть стоит включить в LLM‑префикс дословно (как “policy для генерации кода”), потому что это самая частая зона каскадных отказов:

- **Timeouts**: “Все внешние вызовы (HTTP/gRPC/DB/queue) имеют дедлайн и таймаут; таймауты параметризуются конфигом. Нулевой таймаут запрещён по умолчанию.” citeturn8view1turn7view0turn4search17  
- **Retries**: “Retries только для идемпотентных операций; максимум попыток ограничен; есть backoff + jitter; есть retry budget (ограничение общего объёма retries).” citeturn5search7turn4search6turn4search2turn4search1  
- **Circuit breakers / retry circuit breaking**: “Если используется circuit breaking, он должен в первую очередь ограничивать retries, чтобы объём повторов не взорвался.” citeturn4search2turn11search9  
- **Bulkheads**: “Каждая внешняя зависимость имеет лимит конкурентности (семафор/пул/MaxConnsPerHost).” citeturn11search1turn24view0  
- **Rate limiting / throttling**: “На входе есть rate limiting (token bucket) с корректными кодами ошибок и возможным `Retry-After` для 429.” citeturn11search0turn5search6turn14search2  
- **Backpressure + load shedding**: “Если сервис перегружен — отвечать быстрыми отказами (например 503/429), а не накапливать бесконечные очереди; фиксировать и логировать режим деградации.” citeturn4search1turn4search19turn4search12turn4search0  
- **Graceful shutdown**: “Сервис корректно обрабатывает SIGTERM: перестаёт принимать трафик (readiness), завершает активные запросы в пределах бюджета, закрывает зависимости и выходит.” citeturn24view0turn10search1turn3search1turn3search4  
- **Startup dependencies**: “readiness не должна ‘флапать’ из‑за временных проблем downstream; разделять liveness/readiness/startup probes.” citeturn3search4turn3search0turn4search0  
- **Progressive delivery + evolution**: “Фичи включаются постепенно через feature flags; релизы по возможности canary/blue‑green; крупные миграции — strangler‑подход.” citeturn11search2turn20search4turn20search0turn11search3  

## Concrete good / bad examples и типичные anti-patterns LLM

### Таймауты и корректный HTTP server

**Good**: явные таймауты и shutdown (идея — из net/http документации по Server timeouts и Shutdown). citeturn7view0turn24view0turn10search1  

```go
srv := &http.Server{
	Addr:              cfg.HTTP.Addr,
	Handler:           handler,
	ReadHeaderTimeout: 5 * time.Second,
	IdleTimeout:       60 * time.Second,
	// WriteTimeout: осторожно для streaming; включайте только если у вас JSON API без стриминга.
}

ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()

go func() {
	<-ctx.Done()
	shutdownCtx, cancel := context.WithTimeout(context.Background(), 20*time.Second)
	defer cancel()
	_ = srv.Shutdown(shutdownCtx)
}()

if err := srv.ListenAndServe(); err != nil && !errors.Is(err, http.ErrServerClosed) {
	return err
}
```

**Bad**: `http.ListenAndServe(":8080", handler)` без настройки таймаутов. Нулевые значения таймаутов означают “нет таймаута”, что увеличивает риск удержания соединений. citeturn7view0turn24view0  

### Outgoing HTTP client: reuse, deadline, закрытие body

**Good**: http.Client reuse + per‑request контекст + `defer resp.Body.Close()`. Документация говорит: Client/Transport переиспользовать; Timeout включает чтение body; resp.Body обязан закрыть вызывающий; иначе degrade keep‑alive reuse. citeturn8view1turn19view2turn8view2  

```go
type APIClient struct {
	httpClient *http.Client
	baseURL    string
}

func (c *APIClient) DoSomething(ctx context.Context) error {
	ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
	defer cancel()

	req, err := http.NewRequestWithContext(ctx, http.MethodGet, c.baseURL+"/v1/item", nil)
	if err != nil {
		return err
	}

	resp, err := c.httpClient.Do(req)
	if err != nil {
		return err
	}
	defer resp.Body.Close()

	if resp.StatusCode/100 != 2 {
		return fmt.Errorf("unexpected status: %s", resp.Status)
	}
	return nil
}
```

**Bad**: создавать `http.Client{}` на каждый вызов и не закрывать `resp.Body`. Это конфликтует с рекомендацией reuse Clients и требованием закрывать тело ответа. citeturn8view1turn19view2  

### Строгий JSON и лимит тела запроса

**Good**: `MaxBytesReader` + `DisallowUnknownFields`. citeturn14search0turn14search1  

```go
func decodeJSON(w http.ResponseWriter, r *http.Request, dst any) error {
	r.Body = http.MaxBytesReader(w, r.Body, 1<<20) // 1 MiB

	dec := json.NewDecoder(r.Body)
	dec.DisallowUnknownFields()

	if err := dec.Decode(dst); err != nil {
		return err
	}
	// Запрет “хвоста” (вторая JSON-сущность)
	if dec.More() {
		return fmt.Errorf("unexpected trailing data")
	}
	return nil
}
```

**Bad**: `io.ReadAll(r.Body)` без лимита + `json.Unmarshal` без строгой схемы → риск OOM/ресурсного истощения и “тихих” ошибок. citeturn14search0turn3search7turn14search2  

### Retry с budget и jitter (пример шаблонной утилиты)

Ключ: ретраи без контроля вызывают каскадные сбои; SRE обсуждает, как принимать решение “retry or not” в перегрузке, Envoy рекомендует retry budgets и ограничение retry‑объёма, AWS — backoff+jitter как базовый механизм устойчивости. citeturn4search1turn4search2turn4search6turn4search3  

```go
type RetryConfig struct {
	MaxAttempts int           // включая первую попытку; default: 3
	BaseBackoff time.Duration // default: 50ms
	MaxBackoff  time.Duration // default: 2s
}

func Retry(ctx context.Context, cfg RetryConfig, isRetryable func(error) bool, op func(context.Context) error) error {
	var err error
	for attempt := 1; attempt <= cfg.MaxAttempts; attempt++ {
		if attempt > 1 {
			// full jitter: sleep = rand(0, min(MaxBackoff, BaseBackoff*2^(attempt-2)))
			cap := cfg.BaseBackoff << (attempt - 2)
			if cap > cfg.MaxBackoff {
				cap = cfg.MaxBackoff
			}
			sleep := time.Duration(rand.Int63n(int64(cap) + 1))
			t := time.NewTimer(sleep)
			select {
			case <-ctx.Done():
				t.Stop()
				return ctx.Err()
			case <-t.C:
			}
		}

		err = op(ctx)
		if err == nil {
			return nil
		}
		if !isRetryable(err) {
			return err
		}
	}
	return err
}
```

**Bad**: `for { err = op(); if err==nil {break}; time.Sleep(100*time.Millisecond) }` — бесконечный ретрай без дедлайна и джиттера: повышает риск retry‑storm и cascading failures. citeturn4search0turn5search7turn4search6  

### Rate limiting token bucket

Token bucket документирован в `x/time/rate`: Limiter ограничивает частоту событий, имеет burst size и refill rate; Go Wiki прямо рекомендует token bucket для высоких скоростей. citeturn11search0turn11search4turn14search2  

```go
lim := rate.NewLimiter(100, 200) // 100 rps, burst 200

func rateLimit(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if !lim.Allow() {
			w.Header().Set("Retry-After", "1")
			http.Error(w, "rate limit exceeded", http.StatusTooManyRequests)
			return
		}
		next.ServeHTTP(w, r)
	})
}
```

HTTP 429 и `Retry-After` описаны в RFC 6585. citeturn5search6  

### Типичные LLM-hallucinations / anti-patterns (практический список)

Ниже — ошибки, которые LLM часто генерируют в Go‑сервисах; каждую стоит превратить в “lint‑правило” на ревью:

- **Не передали ctx вниз по стеку / создали context.Background() в середине запроса**. Это нарушает принцип передачи deadlines/cancellation по цепочке. citeturn22view0turn1search1  
- **Сохранили ctx в поле struct**. Go Code Review Comments прямо запрещает это (кроме совместимости интерфейсов). citeturn22view0  
- **Создают `http.Client` на каждый запрос** вместо reuse. net/http явно говорит “Clients should be reused”. citeturn8view1turn8view2  
- **Не закрывают `resp.Body`** → ломают keep‑alive reuse и растят ресурсы. citeturn19view2  
- **Ретраят POST/неидемпотентные операции** без идемпотентности/ключей; RFC 9110 запрещает автоматический retry для non‑idempotent без гарантии семантики. citeturn5search7turn5search3  
- **Делают агрессивные ретраи без budgets/jitter**, создавая положительную обратную связь перегрузки (cascading failures). citeturn4search0turn4search2turn4search6  
- **Смешивают readiness и liveness**: liveness начинает зависеть от downstream (БД/очередь) → Pod рестартится в массовом порядке при проблемах зависимости; Kubernetes определяет разные цели probes и предупреждает о различиях. citeturn3search4turn3search0  
- **Логируют секреты/токены** или кладут их в URL/query; OWASP подчёркивает важность корректного logging и работы с секретами. citeturn12search1turn14search10turn6search3  
- **Используют `math/rand` для “ключей/токенов”**; Go Code Review Comments прямо запрещает для криптографических целей. citeturn22view0

## Отказоустойчивость, деградация и эволюция системы

Ниже — практический “resilience standard” для Go‑template: какие механизмы обязательны и как внедрять без overengineering.

### Почему каскадные сбои — дефолтный риск микросервисов

Каскадный отказ — это отказ, который усиливается во времени через положительную обратную связь: часть системы падает/перегружается → нагрузка перераспределяется → вероятность падения остальных растёт → домино‑эффект. Книга SRE описывает этот механизм как типовой сценарий degradation цепочек. citeturn4search0turn4search1  

Главный усилитель каскадов в микросервисах — **неограниченные retries и очереди**, которые “раздувают” нагрузку на уже проблемный downstream и выжигают ресурсы (потоки, соединения, память). Поэтому retry budgets, rate limiting, bulkheads и load shedding — не “фичи”, а противоядие от положительной обратной связи. citeturn4search2turn4search19turn11search1turn14search2  

### Обязательные механизмы в template и “boring” внедрение

**Timeouts (обязательно)**  
- Каждый входящий запрос должен иметь верхний budget: либо через server timeouts + per‑request deadlines, либо через middleware/handler policy. net/http подчёркивает: `ReadHeaderTimeout` часто предпочтительнее `ReadTimeout` (даёт handler’ам контроль по телу), а `Shutdown(ctx)` позволяет ограничить время остановки. citeturn7view0turn24view0  
- Каждая исходящая операция должна иметь дедлайн через `context.WithTimeout` и/или `http.Client.Timeout`, иначе подвисшие зависимости “прибьют” ваш сервис. AWS Well‑Architected отдельно отмечает важность client timeouts и типовые ошибки (слишком высокие/низкие таймауты, игнорирование retry/CB паттернов). citeturn8view1turn4search17  

**Retries с budget и jitter (обязательно)**  
- Выбирать retryable ошибки только из **узкого списка** (сетевые ошибки, 503/429 при наличии Retry‑After, некоторые `DEADLINE_EXCEEDED`‑сценарии) и **только** для идемпотентных операций; HTTP семантика запрещает автоматические ретраи non‑idempotent без уверенности, что операция идемпотентна. citeturn5search7turn5search6turn5search3  
- Backoff+jitter: AWS описывает backoff+jitter как базовый устойчивый паттерн (и отдельно подчёркивает, что это pillar для remote client libraries), а также как практику “timeouts, retries and backoff with jitter”. citeturn4search3turn4search6

**Circuit breakers (условно‑обязательно, но с дефолтным “breaker на retries”)**

- включить “soft circuit breaking” через ограничения активных retries/коннектов (например лимиты конкурентности и “max outstanding retries”), и только при подтверждённой необходимости добавлять полноценный CB со state machine (closed/open/half‑open). ### Startup dependencies и health checks без overengineering

- Разделять probes:
  - liveness: “контейнер жив и делает прогресс” (например можно детектировать deadlock);  
  - readiness: “готов принимать трафик”;  
  - startup: “дать время на медленный старт, прежде чем лезть в liveness”. Kubernetes официально описывает разницу и назначение каждого probe. citeturn3search4turn3search0  
- Readiness и зависимости:
  - База: readiness включает **только критичные** условия для корректного обслуживания *основного* трафика.  
  - Избегать readiness‑flapping из‑за кратких проблем downstream: иначе при проблеме зависимости вы одновременно снимаете все поды с балансера и делаете ситуацию хуже (положительная обратная связь каскада). Это следует из общей модели cascading failures и overload‑механизмов из SRE. citeturn4search0turn4search1
