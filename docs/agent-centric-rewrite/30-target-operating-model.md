# Target Operating Model (WS-1)

## 1. Назначение

Этот документ фиксирует целевую операционную модель для перехода к agent-centric workflow.
Он отвечает на вопрос: **как система должна работать в целом**, чтобы решать проблему контекстного шума и переусложнения, зафиксированную в:
- `docs/agent-centric-rewrite/00-agent-centric-workflow-vision.md`
- `docs/agent-centric-rewrite/10-system-rewrite-decomposition.md`

## 2. Проблема, которую TOM должен устранить

Текущий pain pattern:
1. Главный поток перегружается линейным проходом множества skills.
2. Контекст разрастается быстрее, чем реальная сложность задачи.
3. Решения плохо читаются постфактум: много промежуточных рассуждений, мало чистого синтеза.

Целевая реакция TOM:
1. Выделить оркестратора как единственную точку финальных решений.
2. Вынести экспертный ресерч в параллельные субагенты с изолированным контекстом.
3. Свести поток в компактные и синтезируемые результаты без принудительного универсального шаблона ответа.

## 3. Design Goals

1. Один универсальный рабочий поток для задач любого масштаба.
2. Минимальный объем главного контекста при сохранении экспертной глубины.
3. Параллельная доменная проработка без потери управляемости.
4. Детерминированный синтез противоречивых рекомендаций.
5. Прозрачный audit trail: понятно, кто дал рекомендацию и почему принято финальное решение.

## 4. Non-Goals

1. Не строить отдельные workflow-типы для “малых/средних/больших” задач.
2. Не превращать TOM в тяжелый gate-процесс.
3. Не убирать skills из системы; меняется их место в архитектуре исполнения.

## 5. Операционные роли

## 5.1 Orchestrator (главный агент)

Ответственность:
1. Держит целостный контекст задачи.
2. Формирует и ограничивает scope.
3. Декомпозирует работу на доменные треки.
4. Запускает субагентов и управляет параллелизмом.
5. Синтезирует результаты и принимает финальные решения.
6. Формирует implementation plan на основе синтезированных решений и обновляет канонические артефакты (`spec.md` и при необходимости `research/*.md`, `plan.md`, `test-plan.md`).
7. На этапе планирования может использовать planning skill в основном потоке (например `go-coder-plan-spec`).
8. На этапе имплементации выполняет изменения кода в основном потоке (в том числе через implementation skill, например `go-coder`).
9. Критически оценивает выводы субагентов по качеству evidence, полноте допущений и применимости к полному контексту задачи.
10. Инициирует перепроверку, если выводы субагентов противоречат друг другу или вызывают сомнения.
11. Оркестрирует параллельные domain review-волны и выполняет cross-domain reconciliation результатов (например security vs performance).
12. Задает output contract для каждой подзадачи субагента в соответствии с целью задачи.

Ограничения:
1. Не делегирует финальное решение субагентам.
2. Не копирует весь сырой ресерч в главный поток.
3. Не использует planning/implementation skills на research-этапах.
4. Не трактует единичный вывод субагента как безусловную истину без синтеза.
5. Не начинает кодинг до фиксации implementation plan.

## 5.2 Subagent (доменный эксперт)

Ответственность:
1. Выполняет узкую подзадачу в своем домене.
2. При необходимости вызывает skills локально.
3. Возвращает результат в формате, заданном оркестратором для конкретной задачи.
4. Работает только в модели `research-only`.

Ограничения:
1. Не меняет глобальный scope.
2. Не принимает финальные продуктовые/архитектурные решения.
3. Не тащит в главный поток лишние промежуточные шаги.
4. Не пишет код и не изменяет файлы репозитория (read-only режим).
5. Не запускает orchestrator-level planning/implementation skills для изменения плана или кода в основном потоке.

## 5.3 Skill

Роль:
1. Локальный инструмент/методика внутри субагента (research first).
2. Источник специализированной компетенции.
3. На этапе implementation planning может использоваться оркестратором в основном потоке как planning-инструмент (например `go-coder-plan-spec`).
4. На этапе имплементации может использоваться оркестратором в основном потоке как implementation-инструмент (например `go-coder`).

Ограничения:
1. Не диктует глобальную оркестрацию.
2. Не обязует главный поток к линейному “skill chain”.

## 6. Матрица прав решений

| Decision Area | Orchestrator | Subagent | Skill |
|---|---|---|---|
| Scope/Non-goals | final owner | input only | no |
| Task decomposition | final owner | input only | no |
| Domain analysis | review owner | primary execution | method support |
| Conflict resolution | final owner | propose options | no |
| Final architecture/product decision | final owner | recommend | no |
| Artifact updates | final owner | draft proposals | no |
| Implementation plan authoring | final owner | forbidden (read-only) | allowed only via orchestrator at planning step |
| Code changes in repository | allowed only at implementation step | forbidden (read-only) | allowed only via orchestrator during implementation |
| Domain code review findings | final adjudicator | primary execution (read-only review) | method support |

## 7. Универсальный жизненный цикл задачи

## Step 1. Intake And Framing

Owner: Orchestrator

Действия:
1. Формирует `Problem`, `Scope`, `Constraints`.
2. Определяет, где нужна экспертная развилка.

Выход:
1. Короткий orchestration brief.

## Step 2. Research Decomposition

Owner: Orchestrator

Действия:
1. Делит задачу на независимые доменные треки.
2. Формирует запросы субагентам.
3. Для high-risk/ambiguous тем может запускать одну и ту же тему в несколько субагентов с разными углами анализа.
4. Для каждого запроса задает адаптивный output contract (required + optional parts).

Выход:
1. Набор subagent tasks (параллельно-исполняемых).

## Step 3. Parallel Domain Research

Owner: Subagents

Действия:
1. Выполняют доменный анализ.
2. При необходимости вызывают skills в режиме анализа.
3. Возвращают ответ по task-level output contract (без forced нерелевантных секций).
4. Не вносят изменения в код/файлы (read-only execution).

Выход:
1. Research packets, готовые к fan-in.

## Step 4. Synthesis And Decision

Owner: Orchestrator

Действия:
1. Объединяет выводы.
2. Разрешает противоречия.
3. Валидирует выводы субагентов и фиксирует устойчивые результаты ресерча в `research/*.md` (по необходимости).
4. Фиксирует финальные решения и открытые вопросы в `spec.md`.
5. При недостаточной уверенности запускает recheck-wave.
6. Может отклонить рекомендацию субагента с явным rationale.

Выход:
1. Обновленный research trail (`research/*.md`, когда нужен) и decision log в `spec.md`.

## Step 5. Implementation Planning

Owner: Orchestrator

Действия:
1. Конвертирует решения и результаты ресерча в исполнимый пошаговый план.
2. Фиксирует порядок реализации, зависимости, checkpoints и минимальную валидацию по шагам.
3. При необходимости использует planning skill в основном потоке (например `go-coder-plan-spec`).
4. Проверяет трассируемость плана к decision log и рискам.

Выход:
1. Заполненный `Implementation Plan` в `spec.md` и/или отдельный `plan.md` при большой реализации.

## Step 6. Implementation And Validation

Owner: Orchestrator (+ исполнители кода)

Действия:
1. Реализация по принятым решениям в основном потоке.
2. При необходимости вызов implementation skill (`go-coder`) из основного потока.
3. Минимально достаточная валидация.
4. Обновление Outcome.

Выход:
1. Завершенный artifact state и проверяемый результат.

## Step 7. Parallel Domain Review (Research-Only)

Owner: Subagents (review domains), orchestrated by Orchestrator

Действия:
1. Запуск параллельных доменных review-субагентов (например security, code quality, performance, reliability).
2. Каждый review-субагент работает в read-only режиме и возвращает findings + рекомендации.
3. Оркестратор выполняет fan-in и выявляет междоменные trade-offs.

Выход:
1. Сводный review-пакет для принятия изменений в коде.

## Step 8. Review Reconciliation And Recheck

Owner: Orchestrator

Действия:
1. Приоритизирует findings и формирует change set без междоменных регрессий.
2. Избегает локально-оптимальных фиксов, ухудшающих другой домен (например security fix, ломающий performance budget).
3. При необходимости запускает повторную review-волну на измененном коде.

Выход:
1. Согласованный набор изменений и подтвержденный post-fix review state.

Этот цикл одинаков для простой и сложной фичи.
Масштабируется количество параллельных треков и глубина исследования, но не тип workflow.

## 8. Правила параллельности

1. Независимые доменные треки запускаются параллельно по умолчанию.
2. Синхронизация выполняется через явный fan-in в точке синтеза.
3. Если вывод субагента меняет общий scope, он не применяется напрямую: сначала синтез у оркестратора.
4. При взаимозависимых рисках допустима вторая волна субагентов после первого fan-in.
5. Для критичных тем допускается intentional overlap: несколько субагентов изучают один и тот же вопрос с разных сторон.
6. Для review-фазы параллельный запуск доменных review-субагентов является рекомендуемым default.

## 9. Context Hygiene Model

Главный контекст должен содержать только:
1. текущую постановку задачи;
2. список финальных/кандидатных решений;
3. открытые вопросы;
4. ссылки на evidence (в том числе на `research/*.md`).

В главный контекст не переносится:
1. полный внутренний reasoning каждого субагента;
2. длинные skill-specific инструкции;
3. повторяющиеся доменные рассуждения без влияния на финальное решение.

## 9.1 Output Contract Policy

1. Нет единого универсального обязательного шаблона для всех subagent-задач.
2. Формат ответа задается оркестратором на уровне конкретного запроса.
3. Контракт должен быть достаточно структурным для синтеза, но не forcing по нерелевантным полям.
4. Субагент может возвращать дополнительные секции, если они повышают качество решения.
5. Если часть контракта неприменима, субагент должен явно это отметить, а не галлюцинировать заполнение.

## 10. Модель синтеза и разрешения конфликтов

Если выводы субагентов конфликтуют, оркестратор применяет единый порядок:
1. Проверка фактического конфликта (терминологический vs реальный).
2. Сравнение предпосылок и ограничений.
3. Проверка качества evidence и уровня confidence по каждой позиции.
4. Дополнительный targeted запрос только по конфликтной зоне.
5. При необходимости запуск второго независимого субагента для same-topic adjudication.
6. Финальный выбор с явным rationale в decision log.
7. Если uncertainty остается — фиксируется open question с owner и unblock condition.

## 10.1 Trust-But-Verify Protocol

Оркестратор обязан:
1. Проверять применимость вывода субагента к полному контексту задачи.
2. Сопоставлять вывод с результатами других субагентов и системными ограничениями.
3. Запускать перепроверку, если evidence слабый/неполный, confidence низкий, есть междоменный конфликт или решение high-impact.
4. Фиксировать случаи отклонения рекомендации субагента с короткой причиной.

## 10.2 Review Reconciliation Protocol

Оркестратор обязан:
1. Рассматривать доменные review-результаты как advisory и потенциально конфликтующие между собой.
2. Собирать минимум один cross-domain pass перед финальным набором фиксов.
3. Проверять, что fix для домена A не нарушает ограничения домена B.
4. При обнаружении конфликта запускать targeted re-review по измененному участку.
5. Закрывать review-цикл только после подтверждения, что критичные междоменные конфликты сняты.

## 11. Операционные инварианты TOM

1. Финальное решение всегда у оркестратора.
2. Главный поток не строится как линейная последовательность skills.
3. Subagent output должен быть синтезируемым и релевантным задаче; фиксированный универсальный формат не обязателен.
4. Параллельность — норма для независимых доменов.
5. Skills используются по требованию, а не по обязательному чеклисту.
6. Subagents всегда read-only и research-only.
7. Перед кодингом обязателен отдельный шаг implementation planning в основном потоке.
8. Изменение кода допускается только в основном потоке на этапе имплементации (при необходимости через `go-coder`).
9. Выводы субагентов имеют advisory-статус до финального синтеза оркестратора.
10. High-impact решения требуют multi-angle research или recheck.
11. Domain code review выполняется через read-only субагентов, а не через изменение кода внутри review-агентов.
12. Финальные post-review изменения утверждаются только после cross-domain reconciliation оркестратора.
13. `spec.md` хранит финальные решения, а подтвержденный исследовательский контекст может храниться в отдельных `research/*.md` без жесткого шаблона.

## 12. Артефактная модель TOM

Основной артефакт:
1. `specs/<feature-id>/spec.md`

Опциональные:
1. `research/*.md` для накопления и переиспользования валидированного ресерча в длинных/неоднозначных задачах.
2. `plan.md` при длинной реализации.
3. `test-plan.md` при насыщенной test-стратегии.

Правило:
1. `spec.md` остается источником истины.
2. `research/*.md` — память проверенного ресерча, не источник финальных решений.
3. Формат `research/*.md` задается по задаче и не фиксируется универсальным контрактом.
4. Остальные файлы — расширения, а не параллельные источники решений.

## 13. Интерфейс к следующим WS

Что TOM передает в `WS-2` (Orchestrator Contract):
1. роль оркестратора,
2. decision rights,
3. synthesis lifecycle,
4. trust-but-verify и adjudication protocol,
5. требование формировать краткие системные инструкции из единого контракта,
6. требование внутренней state-модели оркестратора для mode/state/module gating.

Что TOM передает в `WS-3` (Subagent Contract):
1. роль субагента,
2. требования к adaptive output contract,
3. boundaries использования skills,
4. ожидание, что output пригоден для cross-agent comparison.

## 14. Definition of Done для WS-1

1. Роли `Orchestrator/Subagent/Skill` определены и не конфликтуют.
2. Ownership финальных решений однозначен.
3. Универсальный lifecycle задачи описан end-to-end.
4. Правила параллельной работы и fan-in формализованы.
5. Модель объясняет, как устраняется исходная проблема контекстного загрязнения.

## 15. Open Items (для WS-2/WS-3)

1. Точный контракт запроса к субагенту.
2. Framework адаптивного формата ответа субагента и лимиты объема (текущий формат в `00` — draft, не final).
3. Детальный протокол эскалации и confidence-handling.
4. Формат внутреннего state-object оркестратора (`mode/state/active modules/forbidden actions`) и его интеграция в workflow/runtime.
