# Agent-Centric Workflow Vision

## 1. Контекст

В текущем процессе планирование и спецификация часто строятся вокруг последовательного вызова большого числа `skills` в основном потоке.
Это приводит к росту объема контекста, дублированию рассуждений и усложнению финальных артефактов даже для относительно простых задач.

Цель этого документа: зафиксировать видение перехода от **skill-centric** процесса к **agent-centric** оркестрации, где основной поток остается чистым и управляемым.

## 2. Проблема текущего skill-centric подхода

### 2.1 Основная боль

Когда основной агент последовательно вызывает много `skills`, контекст постепенно засоряется:
- смешиваются промежуточные рассуждения разных доменов;
- теряется ясность, почему конкретные решения были приняты;
- растет вероятность «процесс ради процесса», а не решений по сути фичи.

### 2.2 Последствия

- переусложненные спецификации и workflow-артефакты;
- слабая масштабируемость при длинных цепочках рассуждений;
- ухудшение объяснимости решений для команды;
- повышенная когнитивная нагрузка на главного агента и людей-ревьюеров.

## 3. Целевая модель: Agent-Centric

### 3.1 Базовая идея

Главный агент работает как **оркестратор**:
- держит целостную картину задачи;
- декомпозирует задачу на независимые исследовательские треки;
- вызывает специализированных субагентов;
- принимает итоговые решения на основе их выводов.

Субагенты:
- фокусируются на конкретной экспертной задаче;
- могут использовать `skills` внутри себя по необходимости;
- работают в режиме `research-only/read-only` (без изменения кода и файлов репозитория);
- возвращают короткий структурированный вывод в основной поток.

`Skills` в этой модели не исчезают, но становятся **внутренним инструментарием субагентов**, а не обязательной линейной процедурой в основном контексте.

## 4. Роли в системе

### 4.1 Оркестратор (главный агент)

- отвечает за цель, границы и финальное решение;
- формулирует запросы к субагентам;
- контролирует покрытие рисков;
- синтезирует конфликтующие или частичные выводы;
- переводит результаты ресерча и спецификаций в исполнимый implementation plan перед кодингом;
- применяет критическое мышление и не принимает выводы субагентов как безусловную истину;
- при сомнениях запускает перепроверку (вторая волна субагентов или повторный запрос);
- может опровергнуть вывод отдельного субагента с явным rationale;
- фиксирует итоговую стратегию и план исполнения;
- ведет отдельный validated research trail (`research/*.md`) для длинных задач, чтобы не переисследовать уже закрытые вопросы;
- на этапе планирования в основном потоке может использовать planning skill (например `go-coder-plan-spec`), а на этапе имплементации — implementation skill (например `go-coder`).

### 4.2 Субагенты (доменная экспертиза)

- получают узкий scope (например: reliability, security, API, data, delivery);
- исследуют только свой контур;
- при необходимости сами вызывают соответствующие `skills`;
- выполняют domain review в режиме `research-only` (security review, code quality review, performance review и т.д.);
- используют рабочий draft-формат ответа до финализации контракта в `WS-3`;
- возвращают результат в формате, который задан оркестратором для конкретной задачи;
- типичные элементы ответа (как ориентиры, не жесткий шаблон):
  - вывод,
  - риски,
  - рекомендации,
  - уровень уверенности,
  - что осталось неизвестным.

### 4.3 Skills

- остаются важными как библиотека компетенций;
- применяются локально внутри субагентов;
- не должны диктовать глобальный порядок всей системы.

## 5. Принципы оркестрации

1. **Context isolation**: каждый субагент работает в локальном контексте.
2. **Parallel by default**: независимые исследовательские треки выполняются параллельно.
3. **Synthesis in one place**: финальные решения принимаются только оркестратором.
4. **Adaptive outputs**: оркестратор задает формат ответа под конкретную задачу; субагенты не ограничены одним универсальным шаблоном.
5. **Skills on demand**: skill вызывается только когда реально нужен для конкретной подзадачи.
6. **Trust but verify**: оркестратор проверяет качество и согласованность выводов субагентов перед принятием решений.
7. **Multi-angle research**: критичные темы могут исследоваться несколькими субагентами параллельно с разных сторон.
8. **Parallel domain review**: код ревью выполняется отдельными субагентами по доменам параллельно.
9. **Cross-domain reconciliation**: оркестратор сводит результаты ревью так, чтобы фикс одного домена не ухудшал другой.
10. **Plan before code**: после синтеза оркестратор обязан сформировать implementation plan; кодинг начинается только после этого шага.

## 6. Почему это решает текущую проблему

- основной контекст остается компактным и управляемым;
- снижается дублирование между экспертными проходами;
- проще восстановить причинно-следственную цепочку решений;
- сохраняется доменная глубина без перегруза главного потока;
- повышается прозрачность «кто и зачем дал конкретную рекомендацию».

## 7. Требования к workflow под agent-centric подход

1. Workflow должен описывать:
   - как оркестратор формирует подзадачи;
   - какой формат возврата у субагентов;
   - как оркестратор разрешает противоречия.
2. Workflow не должен навязывать линейный вызов большого числа skills в основном потоке.
3. Параллельность субагентов должна быть нормой, а не исключением.
4. Любой шаг должен быть оправдан риском/ценностью, а не формальным соответствием шаблону.
5. Workflow должен поддерживать перепроверку и adjudication, когда выводы субагентов противоречат друг другу.
6. Workflow должен поддерживать параллельные доменные ревью-волны и последующий reconciliation изменений.
7. Workflow должен содержать отдельный шаг implementation planning в основном потоке между синтезом и кодингом.
8. Workflow должен допускать отдельные `research/*.md` артефакты до/вокруг `spec.md`, где оркестратор фиксирует валидированный ресерч в гибком формате.

## 8. Предварительный подход к формату вывода (draft до WS-3)

Это не финальный контракт и не универсальная обязательная форма.
Это временный подход для согласованности терминов между документами.
Окончательная спецификация будет принята в `WS-3` (`50-subagent-contract.md`) как **framework адаптивного контракта**, а не как один жесткий шаблон.

Базовая идея:
1. Оркестратор задает expected output shape под конкретный subagent task.
2. Субагент обязан вернуть все релевантные для задачи части ответа, но не обязан заполнять нерелевантные поля.
3. Субагент может добавлять дополнительные секции, если это улучшает понимание проблемы.

Примеры полезных секций (не обязательных для каждой задачи):
- `Summary`
- `Risks`
- `Recommendations`
- `Confidence`
- `Open points`

Антипаттерн:
- forcing универсального fixed-шаблона для задач разной природы.

## 9. Ожидаемый эффект

- меньше перегруженных spec-артефактов;
- выше скорость и предсказуемость принятия решений;
- лучшее качество обоснований при меньшем объеме текста;
- меньше повторного ресерча в длинных задачах за счет сохраненного validated research trail;
- проще онбординг для внешнего участника, который читает артефакты постфактум.

## 10. Краткая формулировка позиции

Система должна быть построена вокруг **оркестрации агентов**, а не вокруг линейного прохождения списка `skills`.
`Skills` остаются важными, но используются субагентами как внутренние инструменты в рамках узких задач.
Главный агент сохраняет целостный контекст, принимает финальные решения и держит общий workflow простым и универсальным.
