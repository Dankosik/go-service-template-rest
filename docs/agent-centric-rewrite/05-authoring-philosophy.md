# Agent-Centric Authoring Philosophy

## 1. Зачем нужен этот документ

Этот документ фиксирует стиль проектирования для всей директории `docs/agent-centric-rewrite/`.
Он нужен, чтобы при дальнейшей работе не возвращаться к избыточной формализации и не превращать контракт в набор жестких шаблонов.

## 2. Главная позиция

1. Система должна быть понятной человеку, а не "правильной по форме".
2. Пользовательский вход по умолчанию свободный (как в обычном чате).
3. Оркестратор адаптируется к задаче, а не подгоняет задачу под фиксированный формат.

## 3. Принципы авторинга

1. **Simple-language first**
   - Пишем правила простыми словами и шагами.
   - Сначала объясняем поведение, потом детали.

2. **Protocol over rigid schema**
   - Предпочитаем исполняемый протокол и инварианты, а не обязательные YAML/JSON-формы.
   - Формальные структуры допустимы только там, где они реально повышают надежность исполнения.

3. **Adaptive intake**
   - Оркестратор не требует от пользователя заполнения полей.
   - Оркестратор уточняет только то, что нужно для текущей задачи.

4. **No fake completeness**
   - Нельзя заполнять нерелевантные или неизвестные поля "для полноты".
   - Неизвестное фиксируется как допущение/открытый вопрос.

5. **Minimal sufficient structure**
   - Добавляем только ту структурность, которая помогает синтезу, traceability и качеству решений.
   - Любая избыточная бюрократия удаляется.

6. **Context-driven planning**
   - План не должен быть фиксированным чеклистом для всех задач.
   - Состав плана зависит от контекста (prod/non-prod, риск-профиль, цели пользователя, стадия продукта).
   - Rollback, backward compatibility и rollout-практики включаются только когда они релевантны и/или явно требуются.

7. **Single-source behavior**
   - Контракт должен оставаться единым и непротиворечивым.
   - Не создаем конкурирующие "вторые источники истины" для одних и тех же правил.

8. **Trust-but-verify remains mandatory**
   - Простота формулировок не отменяет качество: conflict resolution, recheck, review reconciliation и planning-before-code остаются обязательными.

## 4. Когда допустима более строгая формализация

Более строгий формат можно вводить только если одновременно выполняются условия:
1. Есть явный machine-interface use case (парсинг, автоматическое исполнение, строгая валидация).
2. Без формализации растет риск реальных ошибок исполнения.
3. Формат не заставляет выдумывать данные, которых нет.
4. У формализации есть измеримый выигрыш в eval/надежности.

Если хотя бы одно условие не выполняется, используем простой текстовый протокол.

## 5. Обязательный self-check для новых документов

Перед финализацией любого нового файла в `docs/agent-centric-rewrite/`:
1. Можно ли понять документ без чтения внутренних шаблонов и схем.
2. Нет ли принуждения пользователя к структурированному вводу.
3. Нет ли мест, где агент "обязан заполнить поля", даже если данные отсутствуют.
4. Сохранился ли фокус на решении задачи, а не на процессной бюрократии.
5. Согласован ли документ с базовыми инвариантами TOM и оркестратора.

## 6. Область действия

Эта философия обязательна для:
1. Дальнейшего проектирования файлов в `docs/agent-centric-rewrite/`.
2. Переписывания `docs/spec-first-workflow.md`.
3. Подготовки системных инструкций в `AGENTS.md` по итогам WS-цепочки.
