# Orchestrator Contract (WS-2)

## 1. Назначение

Этот документ описывает, как оркестратор работает на практике в agent-centric модели.
Цель контракта:
1. Дать понятный и исполняемый протокол без лишней формальности.
2. Сохранить гибкость для разных пользовательских путей.
3. Зафиксировать границы ответственности и понятную последовательность шагов до кодинга.

## 2. Базовая позиция

1. Контракт оркестратора должен быть простым и адаптивным.
2. Пользователь общается свободно; оркестратор не требует от пользователя формального ввода по шаблону.
3. Оркестратор сам нормализует задачу настолько, насколько нужно для решения, но не заполняет нерелевантные поля "для галочки".
4. Любая недостающая информация помечается как допущение или открытый вопрос, а не выдумывается.
5. Подход к формулировкам и степени формализации должен соответствовать `docs/agent-centric-rewrite/05-authoring-philosophy.md`.

## 3. Роль оркестратора

### 3.1 Ответственность

1. Держит целостный контекст задачи.
2. Формирует и ограничивает scope.
3. Декомпозирует работу на доменные треки.
4. Запускает субагентов и управляет параллелизмом.
5. Синтезирует результаты и принимает финальные решения.
6. Формирует implementation plan на основе синтезированных решений и обновляет канонические артефакты (`spec.md` и при необходимости `research/*.md`, `plan.md`, `test-plan.md`).
7. На этапе планирования может использовать planning skill в основном потоке (например `go-coder-plan-spec`).
8. На этапе имплементации выполняет изменения кода в основном потоке (в том числе через implementation skill, например `go-coder`).
9. Критически оценивает выводы субагентов по качеству evidence, полноте допущений и применимости к полному контексту задачи.
10. Инициирует перепроверку, если выводы субагентов противоречат друг другу или вызывают сомнения.
11. Оркестрирует параллельные domain review-волны и выполняет cross-domain reconciliation результатов; набор доменов определяется целью пользователя и контекстом задачи.
12. Задает output contract для каждой подзадачи субагента в соответствии с целью задачи.

### 3.2 Ограничения

1. Не делегирует финальное решение субагентам.
2. Не копирует весь сырой ресерч в главный поток.
3. Не использует planning/implementation skills на research-этапах.
4. Не трактует единичный вывод субагента как безусловную истину без синтеза.
5. Не начинает кодинг до фиксации implementation plan.

## 4. Intake policy (без жесткого шаблона)

Оркестратор перед запуском субагентов должен прояснить минимум:
1. Что именно нужно решить.
2. Что входит в scope и что точно не входит.
3. Какие ограничения и рисковые зоны уже известны.
4. Какие критерии успеха разумно проверять.
5. Каких данных не хватает и какие допущения сейчас приняты.

Правила:
1. Никакого обязательного YAML/JSON-шаблона для пользовательского ввода.
2. Формулировка intake может быть свободной, но должна быть достаточной для декомпозиции.
3. Если неопределенность критична, оркестратор запускает доисследование до принятия решения.

## 5. Постановка задач субагентам

Каждая subagent-задача должна в явном виде содержать:
1. Цель и границы подзадачи.
2. Что важно проверить.
3. Какие ограничения/риски учитывать.
4. Какой тип результата нужен для синтеза (required части + опциональные).
5. Какие признаки требуют ссылок на evidence или retrieval.

Правила:
1. Независимые треки запускаются параллельно по умолчанию.
2. Для high-impact/ambiguous тем используется multi-angle подход (`primary` + `challenger`), если нет явного обоснования single-angle.
3. Субагент получает только релевантный контекст.
4. Субагенты остаются `research-only/read-only`.

## 6. Синтез и принятие решений

### 6.1 Fan-in протокол

Оркестратор:
1. Сводит выводы в сравнимые тезисы.
2. Разводит терминологические расхождения и реальные конфликты.
3. Сравнивает качество evidence, допущения и применимость.
4. Фиксирует выбранное решение и отвергнутые альтернативы.

### 6.2 Tie-break порядок

Если рекомендации конфликтуют, оркестратор действует так:
1. Сначала определяет явный приоритет пользователя и цели задачи (например: performance-first, speed-to-market, reliability-first, security-hardening).
2. Проверяет non-negotiable границы: безопасность, комплаенс и базовая корректность инвариантов не должны нарушаться.
3. Внутри допустимых границ выбирает решение, которое лучше всего соответствует приоритету пользователя.
4. Если явного приоритета от пользователя нет, применяет best-practice baseline для данного типа задачи.
5. Если остаются несколько валидных вариантов, формирует 2-3 опции с trade-offs и рекомендует одну как default.

Жесткий универсальный порядок приоритетов для всех задач не используется.

## 7. Recheck и escalation

Recheck обязателен, если:
1. Решение high-impact, а evidence слабый или неполный.
2. Есть междоменный конфликт без устойчивого победителя.
3. Уверенность в критичном выводе низкая.
4. Есть заметные пробелы в предпосылках.
5. Есть риск междоменной регрессии после proposed fix.

Эскалация может включать:
1. Targeted follow-up по узкому конфликту.
2. Second-opinion в том же домене.
3. Challenger из другого домена.
4. Retrieval-pass по спорному вопросу.
5. Перенос в open question с owner и unblock condition.

## 8. Epistemic policy и override

1. Любой вывод субагента имеет advisory-статус до финального синтеза.
2. Confidence не равен истинности.
3. Отсутствие evidence означает неопределенность, а не подтверждение.
4. Оркестратор может отклонить рекомендацию субагента, но обязан записать rationale и принятые риски.

## 9. Review-reconciliation protocol

Для review-волн оркестратор:
1. Сначала определяет цели ревью из пользовательского запроса и текущего контекста задачи.
2. Формирует набор доменных review-субагентов под конкретную цель, без фиксированного закрытого списка категорий.
3. Запускает выбранные review-субагенты параллельно.
4. Сводит findings в единый список с приоритизацией по severity и blast radius.
5. Проверяет, что fix в одном домене не ломает другой.
6. При конфликте запускает targeted re-review.
7. Закрывает цикл только после снятия критичных междоменных конфликтов.

Примеры возможных доменов ревью:
1. Security, performance, reliability, code quality.
2. Idiomatic Go.
3. SOLID/DRY/KISS и другие инженерные принципы.
4. Любые дополнительные критерии, явно запрошенные пользователем.

## 10. Planning-before-code protocol

В нормальном рабочем ходе кодинг начинается после двух шагов:
1. Синтез завершен.
2. Implementation plan зафиксирован.

План формируется под контекст конкретной задачи и приоритеты пользователя, а не по жесткому универсальному шаблону.

Обязательный минимум плана:
1. Понятная последовательность ближайших шагов.
2. Критерий завершения шага/итерации.

Контекстные элементы (добавляются только когда релевантны):
1. Зависимости и checkpoints.
2. Валидация и quality gates.
3. Трассируемость к решениям и рискам.
4. Rollback/mitigation (если есть production-риск, пользователь этого хочет или контекст требует обратимости).

Пример контекстной адаптации:
1. Для pre-prod/prototype сценариев можно не требовать backward compatibility и rollout-процедуры, если пользователь явно принимает этот риск.

Политика использования skills:
1. Planning skill применяется только на planning-этапе.
2. Implementation skill применяется только на implementation-этапе.
3. На research/synthesis/review этапах planning/implementation skills не используются.

## 11. Артефакты и traceability

1. `spec.md` хранит финальные решения.
2. `research/*.md` хранит validated research context и не заменяет `spec.md`.
3. `plan.md` используется для длинной реализации.
4. `test-plan.md` используется при насыщенной test-стратегии.

Минимальный audit trail:
1. Короткий intake brief.
2. Набор subagent-задач.
3. Decision log.
4. Override log (если был override).
5. Open questions с owner/unblock condition.
6. Фиксация planning-шага перед кодингом.

## 12. Внутренняя state-модель оркестратора

`mode/state` — внутренняя механика оркестратора, не внешний пользовательский ввод.

Базовые состояния:
1. `intake`
2. `research`
3. `synthesis`
4. `planning`
5. `implementation`
6. `review`
7. `reconciliation`
8. `validation`
9. `done`

Переходы:
1. Основной путь: `intake -> research -> synthesis -> planning -> implementation -> review -> reconciliation -> validation -> done`
2. Допустимые циклы:
   - `synthesis -> research` (recheck wave)
   - `reconciliation -> review` (post-fix re-review)
   - `validation -> planning` (если найден плановый gap)

Практика использования:
1. Состояния помогают держать рабочий ритм и не терять этапы.
2. Оркестратор может адаптировать переходы по контексту задачи и приоритетам пользователя.
3. Состояние определяется протоколом оркестратора, а не пользовательским текстом.

## 13. Prompt-quality принципы для будущего AGENTS.md

1. Measurability first: изменения системных инструкций верифицируются через success criteria и eval-набор.
2. Drift control: смена модели/версии идет с regression eval.
3. Context engineering: критичное ставится в начало/конец системного блока.
4. Channel separation: инструкции, данные и примеры логически разделяются.
5. Chaining over monolith: сложные задачи решаются многошаговым процессом, а не one-shot.
6. Tools + validation: структурные результаты проверяются parser/schema перед действием.
7. Retrieval for knowledge: knowledge-intensive утверждения опираются на retrieval.
8. CoT policy: CoT опционален для сложного reasoning и не считается доказательством истинности.
9. Stochasticity management: для детерминированных задач низкая вариативность, для генеративных — контроль через eval.
10. Prompt vs fine-tuning: при чрезмерной длине промптов и наличии данных допускается fine-tuning/PEFT, но без отмены eval-контуров.

## 15. WS-2 Definition of Done

Контракт WS-2 готов, если:
1. Пошагово исполняем оркестратором без неявных решений.
2. Детерминированно обрабатывает конфликтующие subagent-выводы.
3. Формализует trust-but-verify для high-impact решений.
4. Формализует cross-domain reconciliation для review-волн.
5. Закрепляет отдельный implementation planning шаг перед кодингом.
6. Остается простым: без жесткого обязательного шаблона пользовательского ввода.
