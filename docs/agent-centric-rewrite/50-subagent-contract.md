# Subagent Contract (WS-3)

## 1. Назначение

Этот документ фиксирует исполняемый контракт для всех субагентов в agent-centric модели.
Цель контракта:
1. Сделать subagent-выводы совместимыми для fan-in и синтеза оркестратором.
2. Сохранить `research-only/read-only` режим как жесткую границу.
3. Дать адаптивный framework ответа без единого обязательного универсального шаблона.
4. Ограничить контекстный шум в основном потоке.

## 2. Базовая позиция

1. Субагент не владеет финальным решением; он поставляет доменное advisory-мнение.
2. Формат subagent-ответа задается задачей, а не фиксированной глобальной формой.
3. Нельзя выдумывать данные для "полноты": неизвестное фиксируется как допущение или open point.
4. Краткость важна: в основной поток передается только синтезируемый результат, а не длинный внутренний narrative.
5. Субагент не навязывает оркестратору rigid input-формат и принимает задачу в свободной форме.
6. Формулировки и степень формализации должны соответствовать `docs/agent-centric-rewrite/05-authoring-philosophy.md`.

## 3. Роль субагента

### 3.1 Ответственность

1. Выполняет узкую подзадачу в заданных оркестратором границах.
2. Анализирует только релевантный доменный контур и явно отмечает междоменные риски.
3. При необходимости локально использует skills как инструменты анализа.
4. Возвращает результат в task-level формате, заданном оркестратором.
5. Явно отделяет факты, выводы, допущения и открытые вопросы.

### 3.2 Ограничения

1. Не меняет глобальный scope задачи и не переписывает цели оркестратора.
2. Не принимает финальные продуктовые/архитектурные решения.
3. Не пишет код и не изменяет файлы репозитория.
4. Не запускает orchestrator-level planning/implementation path для изменения плана или кода в основном потоке.
5. Не тащит в основной поток сырой развернутый reasoning, если это не было явно запрошено.

## 4. Входной task contract от оркестратора

Источник истины для структуры subagent-задачи: раздел 5 в `docs/agent-centric-rewrite/40-orchestrator-contract.md`.
Субагент обязан быть совместимым с этим контрактом и не вводить более жестких обязательных полей.
`Task contract` может передаваться в свободном тексте.

### 4.1 Минимум, синхронизированный с WS-2

Если оркестратор следует своему протоколу WS-2, subagent-задача содержит:
1. Цель и границы подзадачи.
2. Что важно проверить.
3. Какие ограничения/риски учитывать.
4. Какой тип результата нужен для синтеза (required части + optional части).
5. Какие признаки требуют ссылок на evidence или retrieval.

Субагент не вправе требовать от оркестратора дополнительных обязательных полей сверх этого набора.

### 4.2 Контекстные расширения (опционально)

1. Приоритет пользователя (например `speed-first`, `reliability-first`, `security-hardening`).
2. Список конкретных артефактов/файлов для проверки.
3. Явные tie-break правила для спорной зоны.
4. Ограничение по времени/стоимости доисследования.
5. Режим проработки и предпочтение по краткости (например: "дай быстрый ответ", "копай глубоко", "разбери конфликт точечно").

### 4.3 Политика неполного входа

1. Отсутствие части полей не блокирует запуск subagent-задачи.
2. Субагент сам адаптивно нормализует вход под задачу и контекст.
3. Если критичный параметр отсутствует, субагент запрашивает краткое уточнение у оркестратора или работает с явным допущением.
4. Допущения помечаются явно и привязываются к риску ошибки решения.
5. Субагент не заполняет отсутствующие данные фиктивным содержимым.

## 5. Протокол исполнения субагента

1. Принять и локально нормализовать задачу в рамках доступных `Goal/Scope/Checks`.
2. Выбрать минимальный набор доменных проверок и источников evidence.
3. Выполнить анализ (при необходимости с локальным skill usage).
4. Сжать результат до синтезируемого пакета под `Expected output` (или под базовые ориентиры раздела 6.1, если explicit shape не задан).
5. Перед отправкой пройти self-check (раздел 13).

## 6. Adaptive Output Contract Framework

### 6.1 Базовые ориентиры синтеза

Это не фиксированный шаблон для всех задач, а общий каркас сравнимости.
По умолчанию subagent-ответ стремится включать:
1. `Answer`: основной вывод или рекомендация.
2. `Why`: ключевые основания/evidence.
3. `Risks/Trade-offs`: важные риски и компромиссы.
4. `Confidence`: уровень уверенности и его основания.
5. `Open points / Next checks`: что осталось неопределенным.

### 6.2 Required parts (task-specific)

1. Required части задает оркестратор в конкретной subagent-задаче.
2. Субагент обязан закрыть каждую required часть:
   - дать ответ,
   - или отметить `N/A` с причиной,
   - или пометить как `Open point` с unblock condition.
3. Required части имеют приоритет над "красивой формой" ответа.

### 6.3 Optional sections

Субагент может добавлять секции, только если они реально улучшают решение, например:
1. Альтернативы и их trade-offs.
2. Edge cases.
3. Suggested experiments/rechecks.
4. Влияние на соседние домены.

### 6.4 Политика non-applicable секций

1. Optional секции можно не заполнять, если они нерелевантны.
2. Forced-заполнение нерелевантных пунктов запрещено.
3. Для required, но неприменимых пунктов указывается: `N/A`, причина, и влияет ли это на решение оркестратора.

### 6.5 Режимы ответа (для сопоставимости)

Оркестратор может задавать режим:
1. `research`: рекомендация для решения + риски + уверенность.
2. `review`: findings и рекомендации в read-only режиме.
3. `adjudication`: проверка спорной гипотезы/конфликта и вердикт по конфликтной зоне.

Единственный глобальный rigid-шаблон для всех режимов не вводится.

## 7. Требования к evidence и ссылкам

1. Существенные утверждения должны быть опираемы на проверяемое evidence.
2. Для локальных артефактов указываются путь и позиция (например `file:line`), когда это важно для проверки.
3. Для внешних источников указываются ссылка и дата получения.
4. Факты, интерпретации и допущения должны быть различимы по формулировкам.
5. Отсутствие evidence фиксируется как неопределенность, а не как подтверждение.
6. Запрещены вымышленные ссылки и неподтвержденные "точные факты".

## 8. Confidence и uncertainty policy

1. Для ключевых выводов указывается уровень уверенности (`high`, `medium`, `low`).
2. Субагент кратко объясняет, почему выбран именно этот уровень confidence.
3. При `low confidence` в high-impact зоне субагент обязан рекомендовать recheck/second-opinion.
4. Если вывод чувствителен к допущениям, это явно помечается и передается в open points.

## 9. Политика использования skills внутри субагента

1. Skills внутри субагента опциональны и применяются по необходимости.
2. Выбирается минимально достаточный набор skills, без цепочек "ради процесса".
3. Инструкции skill не отменяют ограничения WS-3 (`research-only/read-only`, без takeover оркестраторских решений).
4. В основной поток не переносится полный лог skill-вызовов, если это не требуется для доказательства.
5. Если релевантный skill отсутствует/устарел, субагент работает best-effort и явно фиксирует ограничение.

## 10. Ожидания по проработке, длине и stop conditions

### 10.1 Режим проработки от оркестратора

1. Оркестратор задает режим обычным языком (например: "дай быстрый ответ", "копай глубоко", "разбери конфликт между A и B").
2. Субагент адаптирует объем анализа под этот режим, не теряя синтезируемость результата.
3. Если режим явно не задан, используется стандартная достаточная проработка.

### 10.2 Политика краткости

1. Ответ должен быть настолько коротким, насколько позволяет корректно закрыть required части.
2. Длинные narrative-фрагменты выносятся в отдельный research-artifact только по явному запросу оркестратора.
3. Повторяющиеся промежуточные рассуждения и вторичные детали в основной поток не отправляются.

### 10.3 Stop conditions

Субагент завершает проход, когда одновременно:
1. Закрыты required части ответа (если они заданы); иначе покрыты базовые ориентиры синтеза из раздела 6.1.
2. Ключевые evidence собраны или явно зафиксированы пробелы.
3. Риски/допущения/uncertainty обозначены.
4. Результат оформлен так, чтобы оркестратор мог напрямую сопоставить его с выводами других субагентов на fan-in этапе.

Если stop conditions не достигаются в заданном бюджете, субагент возвращает частичный результат + эскалацию (раздел 12).

## 11. Read-only enforcement

1. Субагент работает только в режиме `research-only/read-only`.
2. Допустимы действия анализа, не изменяющие репозиторий.
3. Запрещены операции, меняющие код, файлы, git-состояние или основной implementation plan.
4. Любая необходимость write-операции эскалируется оркестратору как отдельное решение.

## 12. Escalation back to orchestrator

Эскалация обязательна, если:
1. Scope двусмысленен настолько, что вывод теряет применимость.
2. Критичный evidence недоступен или противоречив.
3. Конфликт не снимается в рамках доменного контура.
4. Нужна междоменная развилка или новый challenger-проход.
5. Заданный режим проработки/бюджет не позволяет надежно закрыть required части.

Формат эскалации:
1. Что блокирует.
2. Почему это важно для финального решения.
3. Что уже проверено.
4. Какое действие ожидается от оркестратора.

## 13. Минимальный self-check субагента перед отправкой

1. Ответ покрывает все required части или корректно помечает `N/A/Open point`.
2. Ключевые утверждения имеют проверяемое evidence или честно отмечены как допущения.
3. Уровень confidence указан и объяснен для критичных выводов.
4. Текст компактный и пригоден для fan-in.
5. Не нарушены границы read-only и ownership оркестратора.

## 15. WS-3 Definition of Done

Контракт WS-3 готов, если:
1. Любой доменный субагент возвращает совместимый и синтезируемый output.
2. Основной поток не засоряется длинными narrative-выкладками.
3. Режим `research-only/read-only` формализован и исполним.
4. Adaptive output contract framework утвержден вместо единого rigid-шаблона.
5. Зафиксированы требования к evidence, confidence, stop conditions и эскалации.
6. Контракт не конфликтует с `30-target-operating-model.md`, `40-orchestrator-contract.md` и `05-authoring-philosophy.md`.

## 16. Интерфейс к следующим WS

1. Для `WS-4`: этот контракт переносится в обновленный workflow как обязательный протокол subagent-этапов.
2. Для `WS-5`: taxonomy skills должна поддерживать subagent-internal usage и запрет на orchestrator-level skill misuse внутри субагента.
3. Для `WS-6`: портфель субагентов описывается поверх этого контракта (mission/inputs/outputs/default skills).
4. Для шаблонов: `docs/agent-centric-rewrite/templates/subagent-response-template.md` должен быть lightweight helper к этому контракту, а не обязательной универсальной формой.
