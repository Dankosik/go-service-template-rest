# System Rewrite Decomposition (Agent-Centric)

## 1. Цель переписывания

Перестроить текущую spec/workflow систему из skill-centric в agent-centric модель:
- оркестратор держит общий контекст и принимает финальные решения;
- субагенты выполняют узкий ресерч и возвращают синтезируемые выводы по task-level output contract;
- skills используются внутри субагентов по требованию, а не как линейный сценарий главного потока;
- после синтеза оркестратор формирует implementation plan в основном потоке;
- на этапе планирования основной поток может использовать planning skill (`go-coder-plan-spec`);
- на этапе имплементации основной поток может использовать implementation skill (`go-coder`) для изменения кода.
- для длинных/неоднозначных задач оркестратор может вести отдельные `research/*.md` как validated memory, сохраняя `spec.md` источником финальных решений.

## 2. Границы инициативы

В scope:
- workflow-документы и процессная модель;
- контракт оркестратора и субагентов;
- адаптация skills под новую роль;
- миграция артефактов и практики работы команды.

Вне scope (на первый цикл):
- глубокая автоматизация планировщика/рантайма;
- перестройка всех инженерных стандартов репозитория за пределами workflow;
- полная перепись всех существующих legacy-спек в один проход.

## 3. Принципы выполнения

1. Маленькие шаги, без big-bang миграции.
2. Каждый шаг оставляет систему работоспособной.
3. У каждого шага есть проверяемый DoD.
4. Один источник истины на каждом уровне: vision, workflow, contracts, migration.
5. Сначала архитектура процесса, потом массовая адаптация контента.
6. Trust-but-verify: оркестратор обязан критически валидировать выводы субагентов.
7. Simple-language-first authoring: протоколы описываются простыми словами, без обязательных жестких схем пользовательского ввода (см. `docs/agent-centric-rewrite/05-authoring-philosophy.md`).

## WS-1: Target Operating Model (TOM)

Цель:
- формально описать целевую agent-centric операционную модель.

Задачи:
1. Описать роли: `Orchestrator`, `Subagent`, `Skill`.
2. Зафиксировать границы ответственности и ownership решений.
3. Зафиксировать жизненный цикл задачи (инициация -> декомпозиция -> ресерч -> синтез -> implementation planning -> реализация -> валидация).
4. Зафиксировать правила параллелизма и ограничения.

Артефакты:
- `docs/agent-centric-rewrite/30-target-operating-model.md`

DoD:
- внешнему участнику ясно, кто и что делает;
- нет конфликтов ownership между оркестратором и субагентами.

Зависимости:
- WS-0.

## WS-2: Orchestrator Contract

Цель:
- превратить роль оркестратора из абстракции в исполняемый протокол.

Задачи:
1. Определить входной контракт оркестратора (scope, non-goals, risk hotspots).
2. Определить протокол постановки задач субагентам.
3. Определить протокол синтеза (merge, conflict resolution, tie-break rules).
4. Определить протокол эскалации (когда нужно доисследование / когда можно принимать решение).
5. Определить минимальный audit trail решений.
6. Зафиксировать epistemic policy: субагентские выводы advisory, не final truth.
7. Зафиксировать recheck protocol: когда запускать повторный или независимый second-opinion subagent.
8. Зафиксировать правило override: оркестратор может отклонить вывод субагента с явным rationale.
9. Зафиксировать review-reconciliation protocol: как сводить конфликтующие findings из security/quality/performance/reliability review.
10. Зафиксировать planning protocol: как оркестратор превращает решения из синтеза в исполнимый implementation plan до старта кодинга.
11. Зафиксировать policy использования planning skill в основном потоке (на planning-этапе) отдельно от implementation skill.
12. Зафиксировать traceability policy: как связывать финальные решения в `spec.md` с валидированным контекстом из `research/*.md`.
13. Зафиксировать правила формирования кратких системных инструкций из контракта, чтобы избежать prompt overload.
14. Зафиксировать внутреннюю runtime state-модель оркестратора (mode/state/active modules/gates) как часть самого контракта.
15. Зафиксировать enforcement-переходы между состояниями без внешнего пользовательского управления этими полями.

Артефакты:
- `docs/agent-centric-rewrite/40-orchestrator-contract.md`

DoD:
- оркестратор может работать по документу пошагово без неявных решений;
- конфликтующие выводы субагентов обрабатываются детерминированно;
- формализован trust-but-verify протокол для high-impact решений;
- формализован протокол cross-domain reconciliation для code review;
- формализован отдельный шаг implementation planning перед кодингом.
- full contract не используется как always-on prompt blob для LLM; из него формируются короткие системные инструкции.
- mode/state/module gating определяется внутренним протоколом оркестратора, а не внешним пользовательским входом.

Зависимости:
- WS-1.

## WS-3: Subagent Contract

Цель:
- стандартизировать поведение всех субагентов.

Задачи:
1. Определить формат запроса к субагенту на уровне task contract.
2. Спроектировать adaptive output contract framework (вместо единого универсального шаблона), где есть:
   - базовые ориентиры для синтеза;
   - task-specific required parts;
   - optional sections по решению оркестратора;
   - политика для non-applicable секций без forced заполнения.
3. Определить требования к evidence и ссылкам.
4. Определить политику использования skills внутри субагента.
5. Определить SLA по глубине/длине вывода и stop conditions.
6. Зафиксировать read-only правило: субагент не пишет код и не меняет файлы репозитория.

Артефакты:
- `docs/agent-centric-rewrite/50-subagent-contract.md`
- шаблон `docs/agent-centric-rewrite/templates/subagent-response-template.md`

DoD:
- любой доменный субагент возвращает совместимый и синтезируемый output;
- основной поток не засоряется длинными narrative-выкладками;
- для субагентов формализован и соблюдается режим `research-only/read-only`;
- утвержден адаптивный контракт ответа (framework), а не жесткий универсальный шаблон.

Зависимости:
- WS-1.

## WS-5: Skills Refactor Strategy

Цель:
- перевести skills в роль инструментов субагентов.

Задачи:
1. Определить целевую taxonomy skills:
   - `subagent-internal` (по умолчанию),
   - `orchestrator-planning` (для формирования implementation plan в основном потоке, например `go-coder-plan-spec`),
   - `orchestrator-implementation` (для изменения кода в основном потоке, например `go-coder`),
   - `direct-use` (редко, по явному запросу).
2. Сократить обязательные шаблоны output внутри skill-инструкций.
3. Убрать требования, которые принуждают к линейному проходу в главном потоке.
4. Ввести единые lightweight guidance для skill authoring в новой модели.

Артефакты:
- `docs/agent-centric-rewrite/70-skills-refactor-strategy.md`
- обновления в `docs/skills/*` и `.agents/skills/*/SKILL.md` (поэтапно)

DoD:
- skills не навязывают process choreography главному потоку;
- skills можно безопасно вызывать параллельно внутри разных субагентов;
- planning skill usage в основном потоке ограничен этапом implementation planning;
- implementation skill usage в основном потоке ограничен этапом имплементации.

Зависимости:
- WS-2, WS-3, WS-4.

## WS-6: Subagent Portfolio Design

Цель:
- спроектировать набор субагентов для основных доменов.

Задачи:
1. Определить минимальный стартовый набор субагентов (архитектура, API, data, reliability, security, QA, devops).
2. Для каждого субагента определить:
   - миссию,
   - входы,
   - выходы,
   - какие skills обычно вызывает.
3. Зафиксировать правила параллельного запуска и fan-in.
4. Определить шаблоны multi-angle research для одной темы (primary + challenger / same-topic peer review).
5. Выделить review-субагентов как отдельный срез портфеля (security-review, code-quality-review, performance-review и др.) в read-only режиме.

Артефакты:
- `docs/agent-centric-rewrite/80-subagent-portfolio.md`

DoD:
- есть явная карта: задача -> какие субагенты нужны -> что они возвращают;
- для критичных тем формализован сценарий “одна тема, несколько субагентов”;
- для review-фазы описан параллельный запуск нескольких доменных review-субагентов.

Зависимости:
- WS-3, WS-5.

## WS-7: Artifact Model Migration

Цель:
- синхронизировать формат артефактов с новой оркестрацией.

Задачи:
1. Закрепить `spec.md` как canonical decisions artifact.
2. Определить модель `research/*.md` для валидированного ресерча (когда создавать, как линковать в `spec.md`, как избежать дублирования).
3. Зафиксировать, что для `research/*.md` используется гибкий task-driven формат без универсального жесткого шаблона.
4. Определить, когда выносить `plan.md`/`test-plan.md`.
5. Описать миграцию legacy-папок (`00..90`) без обязательного массового переписывания.
6. Убрать дублирующие требования между документами.

Артефакты:
- `docs/agent-centric-rewrite/90-artifact-model-and-migration.md`
- шаблон `docs/agent-centric-rewrite/templates/spec-md-template.md`

DoD:
- новая артефактная модель проста и применима к задачам любого масштаба;
- разделение ролей артефактов однозначно: `research/*.md` (validated research context) vs `spec.md` (final decisions);
- legacy-модель не блокирует новую работу.

Зависимости:
- WS-4.

## WS-8: Orchestration Playbook

Цель:
- дать практический runbook для ежедневной работы оркестратора.

Задачи:
1. Описать канонические сценарии:
   - простая фича,
   - средняя фича с несколькими доменами,
   - крупная cross-cutting фича.
2. Для каждого сценария описать:
   - какой набор субагентов запускать,
   - в каком порядке/параллельно,
   - как делать финальный синтез,
   - как оформлять отдельный implementation planning шаг перед кодингом.
3. Добавить антипаттерны и контрольные вопросы перед завершением.
4. Добавить отдельный сценарий adjudication: противоречивые выводы по одной теме и действия оркестратора.
5. Добавить отдельный сценарий review-wave: параллельные security/quality/performance review и reconciliation post-fix изменений.

Артефакты:
- `docs/agent-centric-rewrite/100-orchestration-playbook.md`

DoD:
- новый участник может пройти задачу end-to-end по playbook без неявной экспертной памяти.

Зависимости:
- WS-2, WS-3, WS-6, WS-7.

## WS-9: Pilot And Validation

Цель:
- проверить модель на реальных задачах и зафиксировать корректировки.

Задачи:
1. Выбрать 2-3 реальные задачи разного масштаба.
2. Прогнать их по новому process loop.
3. Сравнить с baseline:
   - объем артефактов,
   - скорость прохождения,
   - число итераций/переписываний,
   - понятность финальных решений,
   - число междоменных review-конфликтов и доля конфликтов, снятых без регрессий.
4. Зафиксировать корректировки в workflow и contracts.

Артефакты:
- `docs/agent-centric-rewrite/110-pilot-report.md`
- `docs/agent-centric-rewrite/120-post-pilot-adjustments.md`

DoD:
- есть эмпирическое подтверждение, что новая система реально решает исходную проблему.

Зависимости:
- WS-8.

## WS-10: Rollout And Deprecation

Цель:
- мягко перевести репозиторий на новую модель без хаоса.

Задачи:
1. Обновить инструктивные ссылки на новую систему.
2. Пометить legacy-процессы как deprecated.
3. Зафиксировать freeze-период для крупных изменений в старой модели.
4. Определить policy “new work only in new model”.

Артефакты:
- `docs/agent-centric-rewrite/130-rollout-and-deprecation-plan.md`

DoD:
- у команды нет двусмысленности, какой workflow использовать для новой работы.

Зависимости:
- WS-9.

## 6. Контрольные milestones

1. M1: целевая модель и контракты готовы (`WS-1..WS-3`).
2. M2: workflow официально обновлен (`WS-4`).
3. M3: skills/subagents адаптированы на базовом уровне (`WS-5..WS-7`).
4. M4: пилот подтверждает качество (`WS-9`).
5. M5: rollout завершен (`WS-10`).

## 7. Риски и как ими управлять

1. Риск: «документов станет еще больше».
- Контроль: каждый новый документ должен закрывать конкретный WS с явным DoD.

2. Риск: субагенты дают несовместимые выводы.
- Контроль: adaptive output contract + протокол conflict resolution у оркестратора.

3. Риск: долгий переходный период с двумя моделями.
- Контроль: явный rollout/deprecation план и policy для новой работы.

4. Риск: формальная миграция без реального эффекта.
- Контроль: обязательный pilot с измеримыми метриками против baseline.
