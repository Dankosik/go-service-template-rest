# Skill Spec: `go-qa-tester` (Implementation Role)

## 1. Назначение

`go-qa-tester` — implementation skill для тестового слоя в spec-first процессе.

Цель skill:
- реализовать тестовый код строго по утвержденной спецификации (`70-test-plan.md`);
- подтвердить, что инварианты и fail-path контракты реально проверяются в коде;
- не допускать архитектурных или контрактных "додумываний" на фазе реализации.

`go-qa-tester` работает после `Gate G2` и в рамках `Spec Freeze`.

## 2. Ядро Ответственности

`go-qa-tester` отвечает за:
- реализацию `unit/integration/contract` тестов в кодовой базе по `70-test-plan.md`;
- покрытие критичных сценариев из `15-domain-invariants-and-acceptance.md` и `55-reliability-and-resilience.md`;
- корректную организацию тестовой инфраструктуры (фикстуры, test helpers, deterministic setup);
- поддержку стабильности test-suite (детерминизм, изоляция, минимизация flaky-поведения);
- прозрачную проверку качества тестов через команды проекта (`go test`, `go test -race`, `go vet`, дополнительные проверки по плану).

## 3. Роль В Workflow

Основная фаза: `Phase 3. Code-Only Implementation` (`docs/spec-first-workflow.md`).

Операционные правила роли:
- реализует только то, что уже зафиксировано в `70-test-plan.md`;
- не меняет спецификацию и не принимает новые spec-level решения самостоятельно;
- при неоднозначности останавливает реализацию и поднимает `Spec Clarification Request`;
- сохраняет трассируемость между тестами и спецификационными обязательствами.

## 4. In Scope

В scope `go-qa-tester` входит:
- перевод тест-матрицы из `70-test-plan.md` в конкретные test cases;
- реализация негативных и fail-path сценариев (timeouts, retries, cancellation, idempotency), если они есть в плане;
- настройка/обновление тестовых фикстур, моков и вспомогательных инструментов в рамках тестового слоя;
- проверка, что ожидаемое поведение проверяется assert-ами, а не только фактом "тест не упал";
- обеспечение воспроизводимости тестов локально и в CI;
- фиксация тестовых пробелов, если реализация упирается в незакрытый spec gap.

## 5. Out Of Scope

`go-qa-tester` не должен:
- перепроектировать архитектуру, API-контракты, data-модель или security-policy;
- изменять `70-test-plan.md` как источник требований без явного `Spec Reopen`;
- подменять роль `go-qa-review` (ревью и финальная оценка качества тестов);
- решать продуктовые/бизнесовые противоречия вне рамок утвержденной спецификации;
- компенсировать отсутствующие spec-решения "временными" тестовыми интерпретациями.

## 6. Входы И Зависимости

Обязательные входы для работы skill:
- `Gate G2` пройден;
- `70-test-plan.md` утвержден и не содержит открытых блокеров;
- релевантные артефакты согласованы: `15`, `30`, `40`, `50`, `55`, `60`, `90`;
- активен `Spec Freeze` (изменения спецификации только через `Spec Reopen`).

Критичная зависимость:
- качество работы `go-qa-tester` ограничено качеством и полнотой `go-qa-tester-spec` артефактов.

## 7. Ожидаемые Выходы

`go-qa-tester` производит:
- реализованные тестовые файлы/сценарии по матрице `70-test-plan.md`;
- обновленные тестовые фикстуры и вспомогательные тестовые утилиты (при необходимости);
- подтверждение выполнения обязательных quality-check команд;
- список неразрешенных блокеров/противоречий для `Spec Clarification Request` (если обнаружены).

## 8. Интерфейс Со Смежными Ролями

- `go-qa-tester-spec`: источник тестовых обязательств и матрицы сценариев.
- `go-coder`: совместная реализация фазы `Phase 3` без изменения spec intent.
- `go-domain-invariant-spec` и `go-reliability-spec`: источники обязательных сценариев, которые должны быть в тест-коде.
- `go-qa-review`: верифицирует качество реализованных тестов и их соответствие `70-test-plan.md`.

## 9. Definition Of Done

Работа `go-qa-tester` считается завершенной, если:
- все обязательные тесты из `70-test-plan.md` реализованы или явно эскалированы как spec blockers;
- критичные инварианты и reliability fail-path сценарии покрыты в тест-коде;
- нет скрытых архитектурных/spec решений, принятых на стороне тестовой реализации;
- обязательные команды проверки качества выполнены успешно;
- результаты готовы к фазе domain-scoped review (`Gate G4` path).

## 10. Анти-Паттерны

Недопустимо:
- ограничиваться happy-path тестами при наличии fail-path требований;
- писать тесты без связи с `70-test-plan.md`;
- "чинить" пробелы спецификации в тестах без `Spec Clarification Request`;
- создавать нестабильные тесты, завязанные на время/состояние окружения без контроля;
- размывать роль в архитектурный дизайн или в code-review обязанности.
