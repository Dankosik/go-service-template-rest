# Skill Spec: `go-language-simplifier-review` (Domain-Scoped Review)

## 1. Назначение

`go-language-simplifier-review` — review-skill для Phase 4 (`Domain-Scoped Code Review`) в spec-first процессе, который отвечает за снижение когнитивной сложности кода и повышение ясности чтения.

Ценность skill:
- выявляет фрагменты, где код сложно понять, сопровождать и безопасно менять, даже если он формально работает;
- снижает риск дефектов, возникающих из-за перегруженных конструкций, запутанной структуры и неясных имен;
- дает практические и локальные рекомендации по упрощению без несанкционированного redesign.

Workflow-контур (`phases`, `gates`, `Spec Freeze`, `Spec Reopen`) задается в `docs/spec-first-workflow.md`; этот skill отвечает только за readability/simplification-домен.

## 2. Ядро Экспертизы

`go-language-simplifier-review` принимает решения по:
- структурной простоте:
  - глубина вложенности, сложные ветвления, избыточные промежуточные слои;
  - дробление длинных функций на логические шаги с понятной ответственностью;
- когнитивной нагрузке:
  - насколько легко восстановить intent из кода без внешнего контекста;
  - локальность reasoning: можно ли понять участок без перехода по множеству файлов;
- ясности имен:
  - имена типов, функций, переменных и констант выражают смысл, а не только механику;
  - отсутствие вводящих в заблуждение сокращений и неоднозначных терминов;
- выразительности API на уровне кода:
  - сигнатуры и точки вызова читаются как понятные действия;
  - параметры и возвращаемые значения не создают лишнюю ментальную нагрузку;
- устранению incidental complexity:
  - обертки и абстракции без явной пользы;
  - лишняя «церемония», которая скрывает главное поведение.

## 3. Ответственность В Spec-First Workflow

Ключевая роль `go-language-simplifier-review`:
- выполнять domain-scoped review после инвариантно-корректной реализации и до финального design-pass;
- удерживать код в форме, где intent очевиден для следующего инженера без дополнительной реконструкции.

Обязательная ответственность в каждом проходе:
- фиксировать findings только в домене упрощения/читабельности;
- связывать замечание с конкретным `file:line`, риском и минимальным путем упрощения;
- не редактировать spec-артефакты и не вводить новые архитектурные/контрактные решения;
- при конфликте с утвержденной спецификацией инициировать `Spec Reopen` в `reviews/<feature-id>/code-review-log.md`;
- не блокировать merge вкусовыми комментариями без измеримого impact на читаемость и сопровождение.

## 4. Scope Проверок (Что Проверяет Skill)

Обязательный проверочный scope:
- `Control-Flow Simplicity`:
  - уменьшение лишней вложенности и ветвлений;
  - явный happy-path без запутанного прыжка по условиям;
- `Cognitive Complexity`:
  - чтение функции/модуля без постоянного «распутывания» контекста;
  - отсутствие перегруженных блоков, где смешаны разные уровни абстракции;
- `Naming Clarity`:
  - названия отражают смысл и роль;
  - идентификаторы последовательны в рамках домена;
- `Intent Exposure`:
  - бизнес- или техническое намерение не скрыто за шумом реализации;
  - комментарии используются для объяснения причин, а не дублирования очевидного кода;
- `Local Simplification Opportunities`:
  - избыточные helper-цепочки и passthrough-слои;
  - повторяющиеся шаблоны, которые лучше выразить простым и единообразным способом.

## 5. Границы Экспертизы (Out Of Scope)

`go-language-simplifier-review` не подменяет соседние review-роли:
- не проводит idiomatic/correctness review как primary-домен (`go-idiomatic-review`);
- не проводит архитектурный контроль и boundary-compliance как primary-домен (`go-design-review`);
- не валидирует доменные инварианты как primary-домен (`go-domain-invariant-review`);
- не выполняет глубокий performance/concurrency/security/reliability/db-cache аудит как primary-домен.

Также вне scope:
- изменение API-контрактов, спецификаций и архитектурных решений в review-фазе;
- предложения полного redesign, когда достаточно локального упрощения;
- комментарии в стиле «мне так нравится больше» без объективного риска.

## 6. Разграничение С Близкими Ролями

Разделение с `go-idiomatic-review`:
- `go-idiomatic-review` отвечает за idiomatic correctness и Go-дисциплину;
- `go-language-simplifier-review` отвечает за понятность и снижение cognitive load при сохранении утвержденного design intent.

Разделение с `go-design-review`:
- `go-design-review` владеет архитектурной целостностью и drift-контролем;
- `go-language-simplifier-review` владеет локальной простотой кода внутри уже утвержденной архитектуры.

Правило handoff:
- если для исправления замечания нужно менять архитектуру/контракт/spec intent, это не локальное упрощение и должно эскалироваться через `Spec Reopen` и/или в профильный review-skill.

## 7. Deliverables И Формат Результата

Primary deliverable:
- записи в `reviews/<feature-id>/code-review-log.md` в стандартном формате:

```text
[severity] [go-language-simplifier-review] [file:line]
Issue:
Impact:
Suggested fix:
Spec reference:
```

Требования к содержанию findings:
- `Issue`: что именно усложняет чтение/понимание;
- `Impact`: риск ошибки сопровождения, роста change-cost или неверной интерпретации intent;
- `Suggested fix`: минимальный шаг упрощения без выхода за границы домена;
- `Spec reference`: ссылка на релевантный spec-артефакт, если изменение затрагивает утвержденное поведение.

## 8. Severity-Политика Для Упрощения

- `critical`:
  - сложность такова, что высока вероятность неверного изменения критичного поведения;
  - безопасный merge невозможен без упрощения или spec-эскалации.
- `high`:
  - выраженная перегруженность кода с существенным риском ошибок при поддержке;
  - требуется обязательная коррекция до merge.
- `medium`:
  - заметный cognitive overhead без немедленного merge-block, но с повышенным риском регрессий.
- `low`:
  - локальные улучшения ясности и последовательности.

## 9. Definition Of Done Для Прохода Skill

Проход `go-language-simplifier-review` завершен, если:
- проверены изменения на предмет structure simplification, naming clarity и cognitive load;
- все `critical/high` findings оформлены actionably с `file:line`, impact и fix path;
- нет неэскалированных замечаний, требующих архитектурного или контрактного пересмотра;
- вывод остается строго в simplification-domain и не дублирует соседние роли.

## 10. Анти-Паттерны

`go-language-simplifier-review` не должен:
- подменять simplification-review полным design/architecture review;
- давать общие советы без привязки к конкретной точке кода;
- требовать абстракции «на будущее» вместо удаления лишней сложности «сейчас»;
- смешивать semantic correctness с readability, не обозначая handoff;
- принимать spec-level решения напрямую в review-фазе.
